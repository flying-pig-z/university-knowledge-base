## 一、背景：社区的数字统计需求
得物社区有大量的数字统计场景，比如：

+ **用户维度**：点赞数、粉丝数、关注数、发布内容数等
+ **内容维度**：阅读数、分享数、收藏数、评论数等
+ **标签维度**：话题内容数、特效内容数等

这些数字需要**高精确度、高性能查询、支持批量查询**，且场景复杂多样（比如要统计"用户点赞的视频数""用户发布的图文数"等细分场景）。

## 二、之前是怎么样的？
### 老方案：Count数据表 + 缓存
早期采用直接在MySQL中用Count聚合 + Redis缓存的方式。

**优点**：

+ 实现简单，成本低
+ 性能好，绝对精准
+ 适合体量小、单体服务的场景

**但随着业务发展，问题逐渐暴露：**

### 1. **性能瓶颈和稳定性风险**
+ 业务表数据量暴增，分库分表后Count聚合性能变差
+ 统计规则越来越复杂，SQL语句复杂度提升，容易产生慢SQL
+ 计数逻辑和核心业务耦合在一起，一旦出问题会影响核心功能

### 2. **缓存策略问题**
+ **热点穿透**：热门内容/用户的统计数频繁变化，导致缓存不断删除重建，高并发下容易击穿数据库
+ **数据一致性**：缓存更新和MySQL更新无法在同一事务中完成，数据变化频繁时差异会很大

## 三、之后是怎么样的？
### 新方案：独立的计数中心服务
得物团队设计了一套**统一的计数中心**，将计数逻辑从各业务服务中抽离出来，形成独立的平台服务。

### 核心设计思路
#### **写场景（数据更新流程）**
**第一步：数据获取**

+ 不侵入业务代码，通过**订阅业务数据库binlog**的方式获取数据变更
+ 利用得物内部的DTS（数据订阅平台）保证消息不丢失

**第二步：数据处理**

+ 对消息进行幂等校验（防止重复消费）
+ 根据事件类型和规则判断需要变更哪些统计数
+ 例如：用户A关注用户B → A的关注数+1，B的粉丝数+1

**第三步：数据持久化**

+ **数据库**：使用有符号字段存储，允许先减后加的无序消费，保证最终一致性
+ **缓存**：直接操作Redis增减数值并延长有效期

#### **读场景（数据查询流程）**
+ 先查缓存，缓存不存在则查数据库再回写缓存
+ 支持批量跨场景查询
+ 对负数情况做特殊处理

## 四、改造后的收益
### 与老方案对比
| 维度 | 之前 | 之后 |
| --- | --- | --- |
| **架构** | 与业务耦合 | 独立平台服务 |
| **性能** | 复杂SQL易慢查 | 解耦后性能稳定 |
| **稳定性** | 影响核心业务 | 可独立降级 |
| **维护成本** | 每个业务自己维护 | 统一管理，降低成本 |
| **缓存策略** | 热点穿透、数据不一致 | 优化策略，保证最终一致性 |


### 核心优势
1. **解耦核心业务**：计数异常不会影响发布、点赞等核心功能
2. **提升稳定性**：避免复杂统计SQL拖垮数据库
3. **降低维护成本**：统一的计数规则，各业务方无需重复开发
4. **支持降级**：出现问题时可临时降级，展示兜底数据





**简单来说就是：**

**之前**：计数功能散落在各个业务服务中，用Count+缓存实现，简单但扛不住大体量，容易拖垮数据库和核心业务。

**之后**：建设独立的计数中心，通过订阅binlog获取数据、统一处理计数逻辑、独立存储和缓存，实现了性能、稳定性和可维护性的全面提升。

> 目前的常用工具
>
> metrics -- prometheus+ grafana
>
> traces -- opentelemetry / cat
>
> logs -- elk
>

本质上看，企业监控系统的核心问题只有一个：

**当一个复杂系统出问题时，人要尽快知道：有没有问题、问题有多大、发生在哪里、为什么发生。**

而运行中的软件本质上只会留下三类可观测痕迹，所以监控系统自然演化成 `metrics + traces + logs`。

**1. Metrics：回答“整体是否异常”**  
指标是把系统行为压缩成数字时间序列，比如 QPS、错误率、延迟、CPU、连接数。  
它的本质是**低成本、高连续性、可聚合的摘要信息**。

所以 metrics 最适合做：

+ 告警
+ 趋势分析
+ 容量规划
+ SLO/SLI 管理

但它的问题也很明显：  
它告诉你“坏了”，通常不告诉你“哪一次请求坏了、为什么坏了”。

**2. Logs：回答“具体发生了什么”**  
日志是离散事件的文字或结构化记录，本质上是**高信息密度的原始事实**。  
比如某次请求参数异常、SQL 超时、权限校验失败、第三方返回 500。

所以 logs 最适合做：

+ 故障取证
+ 错误细节定位
+ 审计与合规
+ 还原现场

但日志的问题是：  
它信息太多、太碎、太贵，很难直接用来做全局健康判断。你不能靠翻日志来完成全站实时告警。

**3. Traces：回答“问题在链路的哪一段”**  
追踪记录的是一次请求穿过多个服务、组件、数据库时的完整路径和耗时。  
它的本质是**因果关系和调用拓扑的重建**。

所以 traces 最适合做：

+ 分布式请求定位
+ 延迟分解
+ 找到慢点、失败点、调用瓶颈
+ 分析跨服务依赖

它解决的是 metrics 和 logs 之间的空白：  
metrics 太宏观，logs 太局部，traces 负责把“局部事件”串成“全局路径”。

**为什么一定是这三类，不是别的？**  
因为企业排障天然需要三种不同的信息压缩方式：

+ `metrics`：低维摘要，便宜，适合持续监控
+ `logs`：高维事实，详细，适合事后取证
+ `traces`：链路结构，适合定位分布式系统中的因果路径

换句话说，它们分别回答三种不同的问题：

+ `metrics`：现在整体健康吗？
+ `traces`：一次请求卡在哪？
+ `logs`：那里到底发生了什么？

**少一个会怎样？**

+ 只有 metrics：你知道坏了，但不知道为什么。
+ 只有 logs：你能看到细节，但看不到全局，也很难及时告警。
+ 只有 traces：你能看到路径，但缺少足够细节和全局统计。

所以企业不是“习惯上”把监控分成这三部分，而是因为**复杂系统的观测需求本来就天然分裂成摘要、因果、细节三层**。  
`metrics + traces + logs` 本质上是在成本、粒度、定位效率之间达到平衡的最小完备组合。

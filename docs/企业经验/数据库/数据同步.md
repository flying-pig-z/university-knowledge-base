> 这里讨论Mysql之类数据库与其他数据库/异构数据源的同步问题。
>

### 作用
一般用于数据库与不同数据源直接的同步，也可能是不同数据库之间的同步。

有全量定时同步和增量同步两种。

### 全量同步
全量定时同步一般采用datax之类的，每天定时执行

### 增量同步
增量同步最常用的方案是采用数据库订阅，如canal or Flink CDC等，监听binlog，然后将订阅的消息发送给消息队列，消费消息队列完成数据库落库。

如果是大厂会有专门的DTS平台（Data Transporter Service，数据同步服务）同步数据。

另外需要配备对账和手动修复相关代码兜底。



> Canal 的工作原理：Canal 伪装成 MySQL 的从库，通过主从复制协议读取 binlog
>
> Flink CDC是Flink的组件，实时监控binlog上报Flink
>





**为什么不直接使用Canal，要使用MQ：**

不使用MQ：

![](https://cdn.nlark.com/yuque/0/2026/png/42768076/1767594485609-b1cb7490-e85c-420a-a2b2-4a4f2da4d69c.png)

每个 Canal 实例都要与 MySQL 建立独立连接，每个连接都独立请求 binlog 数据，MySQL 要为每个"从库"维护连接、重复发送 binlog

N 个业务 = N 个连接 = N 倍的资源消耗（CPU、内存、网络带宽）

使用MQ后：

![](https://cdn.nlark.com/yuque/0/2026/png/42768076/1767594588858-a87fb548-a4cc-4212-afe2-f6f75d7c6c1e.png)

只需一个 Canal 实例连接数据库，压力从数据库转移到 MQ

### 需要更多的数据怎么办
可以binlog调用别的表join数据，但是可能会有主从延迟，所以可以到达消费者后调用API获取最新的数据。

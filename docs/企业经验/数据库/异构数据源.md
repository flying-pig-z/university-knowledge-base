主要存储Mysql，如果是大规模复杂查询搭配ES，大规模简单宽表查询搭配hbase，大规模离线分析用HIVE，如果小规模简单键值对查询使用Redis缓存。

ES适合复杂查询，因为倒排索引的存在。

HDFS适合数仓分析/较慢的查询，数仓分析一般配合HIVE进行。

> 比如CAT监控系统查询详细日志logview就是先查本地实时数据，如果查不到，再去HDFS查询历史数据。
>

HBASE适合面向用户适合点查，不同于Redis，HBASE适合大数据量。通过一个Key（如用户ID、设备ID）去查询该Key对应的大量属性/字段数据。因为Mysql的话如果宽表的话查询开销太大。

```plain
场景一：用户历史行为记录查询
查询模式：用户查询自己一年前的某次聊天记录、浏览历史、操作日志等。

特点：
数据量巨大，单用户数据也可能很多。
查询模式简单：“获取用户A在某个时间段的记录”。通常以 UserId 作为RowKey的一部分，查询效率极高。
写吞吐量很高，用户每个行为都可能产生一条记录。
通常不需要关联查询。

为什么选HBase：HBase的水平扩展能力可以轻松存储海量日志数据。通过精心设计RowKey（如 userId_reverseTimestamp），可以快速定位到某个用户在某段时间的数据，高效且稳定。MySQL难以承受这种数据量的存储和高速写入。

场景二：用户画像实时查询
查询模式：在推荐、广告等场景，需要实时查询用户的标签。

特点：
数据量巨大（每个用户都有大量标签，如：性别、年龄、兴趣偏好）
查询极其简单：根据 UserId 快速获取该行的所有标签（列）
需要高并发、低延迟的随机读能力
标签数据经常更新

为什么选HBase：HBase的列存储模式非常适合这种“宽表”场景。可以将每个用户的所有标签存储在一行的不同列中，通过 UserId 作为RowKey实现毫秒级的点查。MySQL的行存储模式对于有上千个标签的场景，IO效率低下。
```

另外还有MongoDB，它更像是万金油，集合前面几个存储引擎的优点，但是又都不精通，适合前期使用，也能覆盖大多数场景。



```plain
综合应用：订单系统 Mysql + ES + HBASE的架构

【1】分为订单主表和订单扩展信息表

Mysql单表承受不要大量的数据和并发，首先要进行分库分表，一般通过买家ID和店铺ID分片，适配TOB和TOC

【2】数据量增大，查询维度变多，分库分表之后的跨表查询都会有各种各样的慢查询问题

这时候就需要引入ES来处理复杂查询

【3】ES一般只是同步部分字段，因为全量字段既浪费资源，又导致索引文件过大影响检索效率

所以我们从ES检索完毕之后就需要回来Mysql表的订单扩展表查询
这时候虽然是点查，但是查询效率低，因为需要从多张表获取数据

这时候引入HBASE存储详细信息，以订单号为rowKey，
将订单基本信息及扩展信息全量导入，负责ES检索完毕后HBASE进行数据组装，
同时可以复用于订单导出，数据统计等离线需求上


总结：最终架构变成了Mysql写入，ES负责复杂搜索，HBASE负责最终订单详情组装返回，
同时可以复用于离线分析需求
```

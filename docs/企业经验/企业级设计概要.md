## 多数据源
一般是在MyBatis的配置文件中指定哪个文件夹下的mapper对应什么数据库。

拆分数据库有助于不同数据库负责不同的事，并且可以防止一个数据库过大等等有点

## 数据隔离组件
一个是一个注解，加上注解之后可以拦截这个方法执行的所有SQL语句(SELECT、UPDATE、DELETE)，自动添加 WHERE tenant_id = 'xxx' 条件，这样保证租户之间数据隔离。

当然还有比较重量级的比如一个租户一张表/一个租户一个数据库，但是用户量大的情况下不太现实。

## 冷热分离/数据归档
简单说就是根据更新时间/创建时间，把一些旧的数据归档，比如笔记网站的一些笔记版本，企鹅的很早的聊天记录可以从Mysql中删除，转储到可以存储更多数据的HBASE表。

一般定时扫描删除Mysql中的数据转储到HBASE表。

## 异构数据源
主要存储Mysql，如果是大规模复杂查询搭配ES，大规模简单宽表查询搭配hbase，大规模离线分析用HIVE，如果小规模简单键值对查询使用Redis缓存。

ES适合复杂查询，因为倒排索引的存在。

HDFS适合数仓分析/较慢的查询，数仓分析一般配合HIVE进行。

> 比如CAT监控系统查询详细日志logview就是先查本地实时数据，如果查不到，再去HDFS查询历史数据。
>

HBASE适合面向用户适合点查，不同于Redis，HBASE适合大数据量。通过一个Key（如用户ID、设备ID）去查询该Key对应的大量属性/字段数据。因为Mysql的话如果宽表的话查询开销太大。

```plain
场景一：用户历史行为记录查询（推荐使用 HBase）
查询模式：用户查询自己一年前的某次聊天记录、浏览历史、操作日志等。

特点：
数据量巨大（TB/PB级），单用户数据也可能很多。
查询模式简单：“获取用户A在某个时间段的记录”。通常以 UserId 作为RowKey的一部分，查询效率极高。
写吞吐量很高，用户每个行为都可能产生一条记录。
通常不需要关联查询。

为什么选HBase：HBase的水平扩展能力可以轻松存储海量日志数据。通过精心设计RowKey（如 userId_reverseTimestamp），可以快速定位到某个用户在某段时间的数据，高效且稳定。MySQL难以承受这种数据量的存储和高速写入。

场景二：用户画像实时查询（推荐使用 HBase）
查询模式：在推荐、广告等场景，需要实时查询用户的标签（如：性别、年龄、兴趣偏好）。

特点：
数据量巨大（每个用户都有大量标签）。
查询极其简单：根据 UserId 快速获取该行的所有标签（列）。
需要高并发、低延迟的随机读能力。
标签数据经常更新。

为什么选HBase：HBase的列存储模式非常适合这种“宽表”场景。可以将每个用户的所有标签存储在一行的不同列中，通过 UserId 作为RowKey实现毫秒级的点查。MySQL的行存储模式对于有上千个标签的场景，IO效率低下。
```

另外还有MongoDB，它更像是万金油，集合前面几个存储引擎的优点，但是又都不精通，适合前期使用，也能覆盖大多数场景。

## 数据同步
不同服务之间的资源一般是隔离的，就是一个服务不能调其他服务的数据库。

有全量定时同步和增量同步两种。

全量定时同步一般采用datax之类的，每天定时执行。

增量同步主要是采用数据库订阅，然后将订阅的消息发送给消息队列，消费消息队列完成数据库落库。

另外需要配备对账和手动修复相关代码兜底。

## 灰度
选取部分用户使用某个接口/功能。一般是随机选取，比如要选取百分之十的用户，可以根据userId取余10。

```plain
if ( userId % 10 == 0 ) {
  // 选取十分之一的用户
  doA();
} else {
  doB();
}
```

## 多级缓存

## 聚合写入
像比如点击量 点赞数这类的统计就可以在内存中先进行聚合，然后再写入，减少Mysql的写入量。

### 浮点数底层表示 -- IEEE 754 标准
以 IEEE 754 标准中最常用的双精度浮点数（double precision，64位）为例：

浮点数在内存中的表示由三个部分组成：

```plain
┌───────────────┬──────────────────┬──────────────────────────────────┐
│     符号位     │      指数位      │             尾数位                │
│    1 bit      │     11 bits     │            52 bits               │
└───────────────┴──────────────────┴──────────────────────────────────┘
```

1. **符号位(Sign)** - 1位
    - 0 表示正数
    - 1 表示负数
2. **指数位(Exponent)** - 11位
    - 用于存储科学计数法中的指数部分
    - 使用偏移码表示，偏移值为1023(2^10 - 1)
    - 实际指数 = 存储的值 - 1023
3. **尾数位(Mantissa/Significand)** - 52位
    - 表示科学计数法中的有效数字部分
    - 采用规格化的形式，即二进制小数点左边默认为1
    - 实际精度为53位（隐含的1 + 52位存储）

举个例子，十进制数 5.5 如何存储：

1. 转换为二进制：5.5₁₀ = 101.1₂
2. 规格化：101.1₂ = 1.011₂ × 2²
    - 小数点向左移动，直到小数点左边只有一个1
    - 这个1是隐含的，不需要存储
3. 计算存储格式：
    - 符号位：正数，所以是0
    - 指数：2 + 1023(偏移值) = 1025，二进制为 10000000001
    - 尾数：011后面补0到52位

```plain
0 10000000001 0110000000000000000000000000000000000000000000000000
└┘└──────────┘└───────────────────────────────────────────────────┘
符号   指数                          尾数
```

一些特殊值的表示：

1. **零**：
    - 所有位都是0（+0）
    - 符号位是1，其他都是0（-0）
2. **无穷大**：
    - 指数全1，尾数全0
    - 符号位决定是+∞还是-∞
3. **NaN**（Not a Number）：
    - 指数全1，尾数不全为0

浮点数的这种表示方式让我们能够表示很大范围的数值，但也带来了精度问题，因为：

1. 可表示的数值是离散的，不是连续的
2. 两个相邻可表示数之间的距离不是固定的
3. 某些十进制分数无法用二进制精确表示

### 精度丢失问题
让我通过具体分析 0.1 + 0.2 ≠ 0.3 这个经典问题来解释浮点数为什么会出现精度丢失。

1. 首先看 0.1 如何用二进制表示：
    - 0.1₁₀ = 0.0001100110011001100110011...₂（无限循环）
    - 按照 IEEE 754 双精度格式，需要将其规格化：
    - = 1.1001100110011001100110011...₂ × 2⁻⁴
    - 但由于尾数位只有52位，所以必须截断，无法表示完整的循环
2. 同样 0.2：
    - 0.2₁₀ = 0.001100110011001100110011...₂（无限循环）
    - 规格化后 = 1.1001100110011001100110011...₂ × 2⁻³
    - 同样需要截断
3. 当我们把这两个被截断的数相加时：
    - 首先要对阶（使指数相同）
    - 然后进行尾数相加
    - 最后可能还需要进行规格化

每个步骤都可能引入新的舍入误差：

```plain
0.1 实际存储值 ≈ 0.100000000000000005551115123126...₁₀
0.2 实际存储值 ≈ 0.200000000000000011102230246252...₁₀
它们的和     ≈ 0.300000000000000044408920985006...₁₀
```

这就是为什么 `0.1 + 0.2` 的结果是 `0.30000000000000004`

把这个过程类比成十进制：

+ 就像我们用十进制无法精确表示 1/3（0.333333...）
+ 如果我们只能保留 6 位小数：0.333333
+ 两个这样的数相加：0.333333 + 0.333333 = 0.666666
+ 而实际 1/3 + 1/3 = 2/3 = 0.666666666...
+ 我们在计算过程中损失了精度

这个问题是不可避免的，因为：

1. 计算机使用二进制，某些十进制小数在二进制中是无限循环小数
2. 浮点数的位数是有限的，必须在某处截断
3. 每次运算都可能产生新的舍入误差
4. 这些误差会随着运算累积

这就是为什么在需要精确计算的场景（如金融计算）中，我们通常会：

1. 使用整数计算（比如把金额以分为单位）
2. 使用专门的高精度计算库
3. 或者在支持的语言中使用 BigDecimal 等专门的数据类型

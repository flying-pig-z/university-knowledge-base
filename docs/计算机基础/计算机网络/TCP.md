# 可靠性
## 滑动窗口流量控制
TCP通过**滑动窗口**机制来实现流量控制。

**简单说就是接收方维护一个接收窗口(rwnd)，发送方也维护一个发送窗口(swnd)。**

**接收方每次接收信息都返回的ACK报文中会携带接收窗口的大小，而发送方根据接收窗口的大小动态调整自己发送窗口的大小，使之不能超过接收窗口的大小。**

**当然也会发生接收窗口为0的零窗口情况，这样发送方就没法发送数据了吗？**

**不是的，如果接收窗口为0时，发送方会定期发送窗口探测包获取实时接收窗口的大小。**

这种机制可以防止发送方发送过多数据而导致接收方缓冲区溢出，实现了端到端的流量控制。

## 拥塞控制
拥塞窗口(cwnd)是TCP发送方的一个状态变量。

**拥塞窗口限制了未收到确认的在途报文数量，发送方在一个 RTT 内最多发送拥塞窗口大小的报文段，控制向网络中注入的数据量，防止网络拥塞。**

**通过动态调整拥塞窗口大小来探测网络容量，拥塞窗口增大说明网络状况良好,可以发送更多数据，拥塞窗口减小说明网络可能拥塞,需要降低发送速率。**

发送窗口大小 = min(cwnd, rwnd)

> 接收窗口限制发送速率避免网络拥塞，而拥塞窗口限制发送速率避免接收方缓冲区溢出。
>

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1737705911869-4b2e9286-2f44-42b2-b3ac-fd9dbfa9946a.png)

1. 慢开始

连接建立时,cwnd(拥塞窗口)初始值设为一个较小值(通常为1-4个 MSS)，每收到一个 ACK, cwnd 翻倍增长，直到达到慢启动阈值(ssthresh)或出现拥塞。

2. 拥塞避免

当 cwnd >= ssthresh 时进入拥塞避免阶段，每个 RTT 内 cwnd 只增加1个 MSS，线性增长而不是指数增长,避免过快占用带宽。

3. 超时处理

出现超时时判定为严重拥塞，cwnd 重置为1，设置 ssthresh = cwnd/2，重新进入慢启动阶段。

4. 快重传与快恢复

收到3个重复 ACK 时触发快重传，进入快恢复阶段: 

【1】ssthresh 设为 cwnd 的一半  
【2】cwnd 设为 ssthresh + 3  
【3】每收到重复 ACK, cwnd 加1  
【4】收到新数据的 ACK 后,cwnd 设为 ssthresh

> 重复 ACK 产生的原因：
>
> 1. 数据包丢失： 当接收方收到了不按序到达的数据包时，比如收到了序号为1、2、4、5的包，但是序号3的包丢失了，这时接收方会确认序号2的包（发送 ACK=3），收到序号4的包时，由于序号3还未收到，仍然发送 ACK=3，收到序号5的包时，由于序号3还未收到，继续发送 ACK=3 这样就产生了重复的 ACK，告诉发送方序号3的包可能丢失了。
> 2. 数据包乱序： 网络传输中，后发送的数据包可能先到达，导致数据包乱序。接收方会持续发送期望收到的下一个序号的 ACK，直到收到正确序号的数据包。
>
> 1个或2个重复 ACK 可能是由网络包乱序导致的，不一定是丢包，当收到3个重复 ACK 时，说明很可能确实发生了丢包，此时不等待超时计时器就立即重传，可以提高网络效率。
>
> 【1】ssthresh 设为 cwnd 的一半
>
> 收到3个重复 ACK 表明网络可能出现轻微拥塞，但不像超时那样需要将 cwnd 降到1，因为网络仍然能够传输数据。将 ssthresh 设为当前 cwnd 的一半是一个折中的选择，既降低发送速率又不至于太过保守。
>
> 【2】cwnd 设为 ssthresh + 3
>
> +3 的原因是已经收到3个重复 ACK，说明这3个包已经离开了网络，这些包在网络之外，不会造成网络拥塞，所以可以在新的 ssthresh 基础上额外增加3个包的窗口大小。
>
> 【3】每收到重复 ACK, cwnd 加1
>
> 每个重复 ACK 表明有一个数据包离开了网络，这意味着网络中可以容纳新的数据包，因此可以适当增加 cwnd，为即将发送的新数据包腾出空间。
>
> 【4】收到新数据的 ACK 后, cwnd 设为 ssthresh
>
> 新数据的 ACK 说明丢失的包已经被成功重传，网络已经恢复正常，将 cwnd 设回 ssthresh，开始正常的拥塞避免过程。
>

## AQS协议
ARQ 协议是一种错误控制协议，**用于确保数据在不可靠的网络上可靠传输。**它的基本原理是:接收方通过检错码(如奇偶校验、CRC等)检测到错误时，就要求发送方重传出错的数据帧。

下面主要介绍两种ARQ协议

### 停止等待 ARQ 协议
工作原理：发送方发送一个帧后就停止发送，等待接收方的确认，接收方正确收到帧后发送确认ACK，如果收到错误帧则发送否定确认NAK，发送方收到 ACK 后才发送下一帧，收到 NAK 或超时则重传当前帧。

+ 优点:简单易实现
+ 缺点:信道利用率低，传输效率不高

### 连续 ARQ 协议
工作原理：发送方维护一个发送窗口，可以连续发送多个帧而不需等待确认，接收方维护一个接收窗口，按序接收帧并发送确认，发送方收到确认后窗口向前滑动。

优点：信道利用率高，传输效率好，适合高速网络。

缺点：实现复杂，需要较大的缓冲区。

> 在实际应用中：TCP 使用的是连续 ARQ 的变体，结合了滑动窗口和选择重传机制。
>

## 超时重传
当发送方发送数据之后，它启动一个定时器，等待目的端确认收到这个报文段。接收端实体对已成功收到的包发回一个相应的确认信息（ACK）。如果发送端实体在合理的往返时延（RTT）内未收到确认消息，那么对应的数据包就被假设为已丢失并进行重传。

+ RTT（Round Trip Time）：往返时间，也就是数据包从发出去到收到对应 ACK 的时间。
+ RTO（Retransmission Time Out）：重传超时时间，即从数据发送时刻算起，超过这个时间便执行重传。

RTO 的确定是一个关键问题，因为它直接影响到 TCP 的性能和效率。如果 RTO 设置得太小，会导致不必要的重传，增加网络负担；如果 RTO 设置得太大，会导致数据传输的延迟，降低吞吐量。因此，RTO 应该根据网络的实际状况，动态地进行调整。

RTT 的值会随着网络的波动而变化，所以 TCP 不能直接使用 RTT 作为 RTO。为了动态地调整 RTO，TCP 协议采用了一些算法，如加权移动平均（EWMA）算法，Karn 算法，Jacobson 算法等，这些算法都是根据往返时延（RTT）的测量和变化来估计 RTO 的值。

# TCP TIME WAIT
TCP中的TIME WAIT状态是TCP连接关闭过程中的一个重要状态。当TCP连接主动关闭方发送最后一个ACK包后，该连接会进入TIME WAIT状态，并在这个状态下保持一段时间。

TIME WAIT状态的主要作用:

1. 确保最后一个ACK包能被成功接收。如果这个ACK包丢失，对方会重发FIN包，而处于TIME WAIT状态的一方可以重发ACK包。
2. 防止"延迟的数据包"干扰新连接。TIME WAIT状态确保所有可能延迟的数据包在新连接建立前都已经从网络中消失，避免旧连接的数据包被误认为是新连接的数据包。

TIME WAIT状态虽然有助于确保TCP连接的可靠关闭，但在高并发服务器环境中，大量的TIME WAIT连接会占用系统资源。为了解决这个问题，一些系统提供了配置选项来调整TIME WAIT超时时间或启用TIME WAIT重用。

### 关于默认端口号
HTTP 默认使用 80 端口

HTTPS 默认使用 443 端口

这些端口号只是一个约定，是为了方便：

1. 当你在浏览器输入 `http://example.com` 时，不需要指定端口，浏览器会自动连接到 80 端口
2. 当你输入 `https://example.com` 时，浏览器会自动连接到 443 端口

但实际上 HTTP 服务可以运行在任何端口上：

+ 比如常见的开发服务器用 8080、3000 等端口
+ Nginx、Apache 等 web 服务器也可以配置任意端口
+ 只是需要在访问时明确指定端口号，如 `http://example.com:8080`

## 传输层
HTTP具体发送过程是怎么样的呢？这个就涉及到传输层。

HTTP在实际传输数据之前要进行TCP三次握手建立连接，传输完数据之后要进行TCP四次挥手断开连接。

```bash
客户端                                    服务端
   |                                        |
   |  1.DNS解析获取服务器IP                  |
   |  2.TCP三次握手建立连接                  | 
   |     --- SYN -->                        |
   |     <-- SYN+ACK ---                    |
   |     --- ACK -->                        |
   |                                        |
   |  3.发送HTTP请求                        |
   |     --- HTTP Request --->              |
   |     <-- HTTP Response ---              |
   |                                        |
   |  4.TCP四次挥手断开连接                  |
   |     --- FIN -->                        |
   |     <-- ACK ---                        |
   |     <-- FIN ---                        |
   |     --- ACK -->                        |
```

## 网络层
### 在开始前 -- DHCP
在开始发送请求前，首先你的主机要先连接网络，此时你的主机在接入网络或者热点的时候之后会通过DHCP自动配置下面信息：

1. IP地址
2. 子网掩码
3. 默认网关（会影响路由表）
4. DNS服务器地址

### 具体流程
那网络层具体怎么发送数据呢？

1. 先去路由表找到目的IP对应的网络，对应的接口
2. 通过接口IP从ARP缓存中或者发起ARP广播获取到接口对应的MAC地址
3. 发送请求
4. 由网关负责接着转发请求，具体流程同上，经过一跳一跳的传播后（不同路由器中间可能还有无路由功能的交换机）最终请求到服务器。

> 路由器和交换机：
>
> + 路由器的完整流程：上面讲的。
> + 交换机完整流程：交换机A收到帧，学习源MAC地址；查MAC地址表，找到下一个转发端口；不改变MAC地址，直接转发；交换机B、C重复相同过程；最终帧到达目标主机。
> + 路由器主要完成网络间(例如公网和内网间)的转发,工作在网络层；交换机主要完成同一局域网内的数据转发,工作在数据链路层。虽然路由器内置了交换功能,但端口数量有限且性能较弱。
> + 网络拓扑设计 典型的企业网络拓扑：Internet -> 路由器 -> 核心交换机 -> 接入层交换机 -> 终端设备
> + 同一局域网内，MAC地址保持不变，只有经过路由器才会改变MAC地址，交换机通过MAC地址表进行转发决策。
>

如果是从私网到公网的路由器，还会进行NAT转换，将私有IP+端口转化为公有IP+端口（不同私网之间可能也会），从而使用公网身份访问服务器。

使用公网IP+端口，也使得服务器返回数据的时候也能访问到我们客户端。

## 数据链路层和物理层
数据链路层进行如下操作：

1. 可靠性：通过CRC校验和冲突处理
2. 效率：CSMA/CD机制避免冲突
3. 有序性：帧序列号和确认机制
4. 流量控制：避免接收方缓冲区溢出

最后通过网卡，并且通过物理层的多种多样的传输介质：双绞线，同轴电缆，光纤，无线电波(用于WiFi)传入给对应的主机。

对应的主机再进行层层接收和解析，解析后返回数据，我们将返回的数据解析渲染在网页上，一次HTTP操作就完成了。

### 层层封装

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1737341511610-7b115d5a-e7b1-40e2-ab2e-fa54eda759b0.png)

1. 应用层封装（定义应用之间通信的协议 --> 报文）

原始数据首先被添加应用层协议头，例如HTTP协议会添加HTTP头部信息，包含请求方法、URL、请求头等，形成应用层PDU(协议数据单元)。

2. 传输层封装（加端口号等 --> 段）

接收应用层数据后添加TCP或UDP头部。

TCP头部包含：源端口、目标端口、序列号、确认号、窗口大小等，UDP头部较简单：只有源端口、目标端口、长度和校验和，形成段(Segment)。

3. 网络层封装（加IP地址等 --> 包）

添加IP头部信息，IP头部包含：源IP地址、目标IP地址、TTL、协议号、分片信息等，形成包(Packet)。

4. 数据链路层封装（加Mac地址等 ---> 帧）

添加MAC头部和尾部，头部包含：源MAC地址、目标MAC地址、类型/长度字段，尾部添加FCS(帧校验序列)用于错误检测，形成帧(Frame)。

5. 物理层转换（比特传输【物理层面】）

将帧转换为比特流，通过物理媒介(如网线)传输。

**注：报文，段，包，帧只是数据在不同层的不同名字，本质上都是数据，本质上都是字符串。**

### 数据分片
1. 应用层

不负责分片，但会根据应用协议(如HTTP)对数据进行格式化和封装，产生原始数据并传递给传输层。

2. 传输层

TCP会根据MSS(最大报文段长度)对数据进行分段，典型的MSS值约为1460字节(以太网环境)。

UDP一般不分片,但如果数据超过MTU会依赖IP层分片

3. 网络层

IP协议根据MTU(最大传输单元)对数据包进行分片，以太网的MTU通常是1500字节，每个分片都会携带分片信息(如偏移量)以便目的地重组。

4. 数据链路层

将数据分成帧(Frame)，帧的大小取决于具体的链路层协议，例如以太网帧最大为1518字节(包含帧头和帧尾)。

5. 物理层

不负责分片，将数据转换成比特流在物理媒介上传输。

总结来说,主要的分片/分段发生在:

+ 传输层(TCP分段)
+ 网络层(IP分片)
+ 数据链路层(帧的封装)

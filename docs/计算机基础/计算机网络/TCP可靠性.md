## 滑动窗口流量控制
TCP通过**滑动窗口**机制来实现流量控制。

**简单说就是接收方维护一个接收窗口(rwnd)，发送方也维护一个发送窗口(swnd)。**

**接收方每次接收信息都返回的ACK报文中会携带接收窗口的大小，而发送方根据接收窗口的大小动态调整自己发送窗口的大小，使之不能超过接收窗口的大小。**

**当然也会发生接收窗口为0的零窗口情况，这样发送方就没法发送数据了吗？**

**不是的，如果接收窗口为0时，发送方会定期发送窗口探测包获取实时接收窗口的大小。**

这种机制可以防止发送方发送过多数据而导致接收方缓冲区溢出，实现了端到端的流量控制。

## 拥塞控制
拥塞窗口(cwnd)是TCP发送方的一个状态变量。

拥塞窗口限制了未收到确认的在途报文数量，发送方在一个 RTT 内最多发送拥塞窗口大小的报文段，控制向网络中注入的数据量，防止网络拥塞。

通过动态调整拥塞窗口大小来探测网络容量，拥塞窗口增大说明网络状况良好,可以发送更多数据，拥塞窗口减小说明网络可能拥塞,需要降低发送速率。

发送窗口大小 = min(cwnd, rwnd)

> 接收窗口限制发送速率避免网络拥塞，而拥塞窗口限制发送速率避免接收方缓冲区溢出。
>

1. 慢开始

连接建立时,cwnd(拥塞窗口)初始值设为一个较小值(通常为1-4个 MSS)，每收到一个 ACK, cwnd 翻倍增长，直到达到慢启动阈值(ssthresh)或出现拥塞。

2. 拥塞避免

当 cwnd >= ssthresh 时进入拥塞避免阶段，每个 RTT 内 cwnd 只增加1个 MSS，线性增长而不是指数增长,避免过快占用带宽。

3. 超时处理

出现超时时判定为严重拥塞，cwnd 重置为1，设置 ssthresh = cwnd/2，重新进入慢启动阶段。

4. 快重传与快恢复

收到3个重复 ACK 时触发快重传，进入快恢复阶段: 

【1】ssthresh 设为 cwnd 的一半  
【2】cwnd 设为 ssthresh + 3  
【3】每收到重复 ACK, cwnd 加1  
【4】收到新数据的 ACK 后,cwnd 设为 ssthresh

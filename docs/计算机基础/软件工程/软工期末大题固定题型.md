## 程序流程
### 程序流程控制图
#### <font style="color:rgb(0, 0, 0);">控制流图概念</font>
<font style="color:rgb(0, 0, 0);">控制流图(Control Flow Graph, CFG)也叫控制流程图，是一个过程或程序的抽象表现，是用在编译器中的一个抽象数据结构，由编译器在内部维护，代表了一个程序执行过程中会遍历到的所有路径。它用图的形式表示一个过程内所有基本块执行的可能流向, 也能反映一个过程的实时执行过程</font>

<font style="color:rgb(0, 0, 0);">个人理解：代码中所有可行路径构成的有向控制流绘制的图</font>

#### <font style="color:rgb(0, 0, 0);">控制流图组成</font>
**<font style="color:rgba(0, 0, 0, 0.87);">圆</font>**<font style="color:rgba(0, 0, 0, 0.87);">: 流图结点，表示一个或多个过程语句。</font>

+ <font style="color:rgba(0, 0, 0, 0.87);">处理框序列和一个菱形判定框可映射为单个结点(相对于流程图)</font>
+ **<font style="color:rgba(0, 0, 0, 0.87);">合并顺序执行的语句, 分离判定语句的不同条件</font>**

**<font style="color:rgba(0, 0, 0, 0.87);">箭头</font>**<font style="color:rgba(0, 0, 0, 0.87);">: 边或连接，表示控制流，一条边必须终于一个节点，即使该结点并不代表任何过程语句。</font>

### 环路复杂度
 V(G) = 边数(E) - 节点数(N) + 2  

         = 判断条件数量 + 1

### 给出独立路径
独立路径，每次判断都会产生不同选择，每个选择会产生不同的路。每条路就是一个路径。

一次改变一个点，最后所有线药全走过

独立路径数等于环路复杂度。

### 测试结点
测试结点是那些决定程序控制流走向的节点，简单说就是决策结点，一般是条件表达式，有多个出口。

### UML用例图
教程：[https://www.bilibili.com/video/BV1qN41177fw/?vd_source=d171ed648157798a725c9e901af91f9e](https://www.bilibili.com/video/BV1qN41177fw/?vd_source=d171ed648157798a725c9e901af91f9e)

用例图有四个元素 -- 系统（矩形），参与者（小人），用例（椭圆形）和关系（实现或虚线）

#### 画系统
先确定系统，用一个矩形表示，并将系统的名称放在顶部，这个矩形定义此系统的范围。

矩形内的任何事情，都会在应用中出现；矩形外的任何事情，都不会在应用中出现。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735122117269-354676fe-7abc-4b86-84a6-68d0db6ba4a0.png)

#### 画参与者元素
确定参与者，用小人表示。参与者系统外部对象，需要放在系统之外。

分为主要参与者和次要参与者。主要参与者做出动作之后，次要参与者才会响应。

主要参与者画在系统的左边，次要参与者画在系统的右边。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735124746752-1056634d-a732-4285-9d42-b279b37eb25c.png)

#### 画用例
用例就是用来描述系统功能，用椭圆形来表示，放置在矩形内，是发生在系统内的动作。

我们将用一些用例来表述每个动作，每个用例都是一个动词，最好是将用例按逻辑顺序排列。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735124834608-8300ec4c-75ed-412f-8fc8-539472c78ace.png)

#### 画关系
最后一个元素是关系，根据参与者的定义，每个参与者必须至少与系统的一个用例交互。

**【1】参与者与用例之间的关系 -- 关联**

在参与者和用例之间画一条实线，来表示这种关系，这种关系叫做关联。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735124516262-43a016ba-4312-4f7a-a102-ac6fb2740e31.png)

**【2】用例之间的关系 -- 包含，扩展和泛化**

除了关联，还有其他三种其他类型的关系。有包含，扩展和泛化。

这三种关系是用例之间的关系，而不是用例与参与者的。

包含关系显示和基本关系和包含关系之间的依赖关系，每次执行基本用例时，包含用例也会执行。当有包含关系时，你需要绘制一条带有箭头的虚线，指向包含的用例，并且在虚线上写上<<include>>。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735124442857-1ed57330-94b3-4c22-b43c-53c69ff6166f.png)

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735123355607-65b4629e-6c11-4b4f-b53b-91b5d0030eee.png)

扩展关系有个基本用例和一个拓展用例，当执行基本用例时，拓展用例不一定会触发，只有在满足某些条件时才会除非，当有拓展关系时，使用一个带有箭头的虚线从拓展用例指向基本用例，并在虚线上写上<<extend>>。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735124465395-7e0a11ee-6483-41f8-91cd-500857cfc0e9.png)

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735123557563-8bcde100-77ac-4efc-8ef9-0fd55a62e977.png)

最后一种关系时泛化，也称为继承。如果两个用例，子用例有着父用例的功能，并且子用例还具有特有的功能。为了描述这种关系，我们从子用例开始，画箭头指向父用例。

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735124423295-3f8d8e9a-dd0a-4aba-a0c0-8f6e91edd708.png)

#### 类之间的关系
![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735129272411-c17e9b1a-0153-4492-8d45-d860ef7c5034.png)

> 从上到下是继承 聚合 关联 组合
>

继承 -- 空心三角形，由子类指向父类

关联 -- 直线

关联 -- 空心菱形， 整体指向部分  

组合 -- 实心菱形， 整体指向部分  

**【1】继承**

+ 子类 获得了 父类 的属性和方法
+ 这是一种 is-a 的关系，比如 Dog 是一个 Animal

```java
// 1. 继承关系（Inheritance）
// 父类
class Animal {
    protected String name;
    
    public Animal(String name) {
        this.name = name;
    }
    
    public void eat() {
        System.out.println(name + " is eating");
    }
}

// 子类继承父类
class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }
    
    public void bark() {
        System.out.println(name + " is barking");
    }
}
```

**【2】关联**

+ 两个类可以独立存在，并且生命周期没有关联
+ 这是一种较弱的关系，比如下面学生可以换老师，老师也可以教不同的学生

```java
// 2. 关联关系（Association）
class Student {
    private String name;
    private Teacher teacher; // 关联关系
    
    public Student(String name) {
        this.name = name;
    }
    
    public void setTeacher(Teacher teacher) {
        this.teacher = teacher;
    }
}
class Teacher {
    private String name;
    
    public Teacher(String name) {
        this.name = name;
    }
}

```

**【3】聚合**

+ 整体和部分之间是可分离的，部分可以属于多个整体
+ 比如员工可以属于不同部门，部门解散后员工依然存在

```java
class Department {
    private String name;
    private List<Employee> employees; // 聚合关系
    
    public Department(String name) {
        this.name = name;
        this.employees = new ArrayList<>();
    }
    
    public void addEmployee(Employee employee) {
        employees.add(employee);
    }
    
    public void removeEmployee(Employee employee) {
        employees.remove(employee);
    }
}

class Employee {
    private String name;
    
    public Employee(String name) {
        this.name = name;
    }
}
```

**【4】组合**

+ **这是最强的关系，部分不能脱离整体而存在，生命周期一致**
+ 比如汽车和发动机，发动机是在创建汽车时一起创建的，销毁汽车时发动机也会被销毁

```java
// 4. 组合关系（Composition）
class Car {
    private Engine engine; // 组合关系
    
    public Car() {
        // 创建Car时必须创建Engine，它们生命周期一致
        this.engine = new Engine();
    }
}

class Engine {
    public Engine() {
        // 发动机的构造函数
    }
    
    public void start() {
        System.out.println("Engine started");
    }
}
```

这些关系的强弱程度：组合 > 聚合 > 关联 > 继承

需要注意的是，在实际开发中，这些关系的选择要基于实际的业务场景和设计需求。比如：

+ 如果两个类有共同的属性和行为，考虑使用继承
+ 如果两个类之间是松散的关联，使用关联关系
+ 如果是整体与部分的关系，且部分可以独立存在，使用聚合
+ 如果部分完全依赖于整体的生命周期，使用组合

> 注：如果一个类是抽象类，可以将类名变成斜体或者用《》拓号起来
>
> ![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735129088473-07361dab-98ba-412e-bd57-3e80b97d3253.png)
>

> 继承和组合都比较好区分，但是关联和聚合就不是那么好区分了。下面是它们的差别：
>
> + 关联是一种较松散的关系,表示两个类之间有联系,但这种联系并不强调整体与部分。表示"有一个"(has-a)关系,比如学生有一个老师。  
关联适用于描述业务功能上的互相协作关系，例如: 订单和客户、学生和课程，这些对象之间是互相独立的业务实体。
> + 聚合特指整体与部分的关系,虽然部分可以独立存在,但语义上更强调"属于"的概念。表示"整体-部分"(whole-part)的关系,比如部门包含员工,公司包含部门。  
聚合适用于描述整体包含部分的关系，例如: 购物车和商品、班级和学生，虽然部分可以独立存在,但在业务上是有明确的从属关系。
>
> 在代码中，就比如上面的学生和教师，如果你在学生中加入教师对象，那就是关联关系。如果你在教师中加入学生集合，那就是组合关系。
>
> 关联关系和聚合关系映射到生活中，你和你的朋友是关联关系,你们之间有联系但都是独立的个体。足球队和队员是聚合关系,队员是球队的一部分,但队员可以换队。

## 模型
### 增量模型
+ **特点**：将产品开发分为多个小的增量，每个增量都是一个可交付的子功能或部分产品，能快速提供可用的部分功能，同时继续开发后续增量。
+ **优点**：可以快速交付部分功能满足用户需求，适合时间紧迫的项目。
+ **适用场景**：时间紧迫、需求可以逐步实现的项目。

### 螺旋模型
+ **特点**：基于迭代的模型，将项目分为多个阶段，每个阶段按风险评估进行开发，适合复杂和需求可能变化的项目。
+ **优点**：强调风险控制，适合需求不明确、风险较高的项目。
+ **缺点**：开发周期较长，难以快速交付成果。

### 瀑布模型
+ **特点**：按严格的顺序开发，从需求分析到设计、实现、测试、交付，每个阶段完成后才能进入下一个阶段。
+ **优点**：流程清晰，适合需求明确、变动较少的项目。
+ **缺点**：整个开发周期较长，无法快速交付成果。
+ **本题适用性**：不适合“时间迫切”的项目。

### 原型模型
+ **特点**：快速开发一个原型以确认用户需求，然后基于原型进行进一步开发。
+ **优点**：适合需求不明确的项目，用户可以通过原型及时反馈需求变化。
+ **缺点**：重点是需求验证，完整产品的开发进度可能会延后。

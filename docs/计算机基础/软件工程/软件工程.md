## 内聚性和耦合性
### 内聚性
#### <font style="color:rgba(0, 0, 0, 0.87);">概念</font>
<font style="color:rgba(0, 0, 0, 0.87);">一个模块内部各个元素彼此结合的紧密程度，一个内聚的模块应该只完成一件事情。应该避免“分裂型”构件。</font>

#### <font style="color:rgba(0, 0, 0, 0.87);">内聚的七个层次</font>
![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735095943463-c428fe2c-eca9-4ba7-9669-252dd7caf8fd.png)

1. **巧合内聚**

**模块中的各个元素之间没有任何有意义的关系,只是被随意地放在一起。**

```java
class Utilities {
    // 这些方法之间没有任何逻辑关系
    public static void printLog(String message) { ... }
    public static int calculateAge(Date birthDate) { ... }
    public static void formatHardDisk() { ... }
    public static String encryptPassword(String pwd) { ... }
}
```

2. **逻辑内聚**

将执行类似操作但实际上不相关的代码组合在一起。这些操作可能共享某些逻辑,但处理的是不同的数据。由控制型参数来确定执行哪一种功能 (选择其中一个功能，内聚性不强)。

```java
class InputProcessor {
    // 虽然都是处理输入,但处理的是不同类型的数据
    public void processKeyboardInput() { ... }
    public void processMouseInput() { ... }
    public void processFileInput() { ... }
    public void processNetworkInput() { ... }
}
```

3. **时间内聚**

模块中的元素在同一时间被执行,但可能执行不相关的操作。

```java
class SystemStartup {
    public void initialize() {
        loadConfigurations();
        initializeDatabase();
        startLogging();
        checkSystemResources();
    }
}
```

4. **过程内聚**

**模块中的元素按特定顺序执行,但可能处理不同的数据或执行不相关的操作。**

```java
class OrderProcessor {
    public void processOrder() {
        validateOrder();
        calculatePrice();
        updateInventory();
        generateInvoice();
        sendConfirmationEmail();
    }
}
```

5. **通信内聚**

模块中的元素操作相同的数据集,但执行不同的操作。

```java
class Customer {
    private CustomerData customerData;
    
    public void updateCustomerInfo() { ... }
    public void displayCustomerDetails() { ... }
    public void calculateCustomerDiscount() { ... }
    public void validateCustomerCredit() { ... }
}
```

6. **顺序内聚**

**在过程内聚的基础上，模块中的元素之间存在数据流,一个元素的输出作为另一个元素的输入。**

```java
class TextProcessor {
    public String processText(String text) {
        String lowercase = convertToLowerCase(text);
        String noWhitespace = removeExtraWhitespace(lowercase);
        String normalized = normalizeCharacters(noWhitespace);
        return normalized;
    }
}
```

7. **功能内聚**

这是最强的内聚形式。模块中的所有元素都共同完成单一的、定义明确的任务。(符合类的单一职责原则)

```java
class PasswordValidator {
    public boolean validatePassword(String password) {
        return checkLength(password) &&
               containsUpperCase(password) &&
               containsLowerCase(password) &&
               containsDigit(password) &&
               containsSpecialChar(password);
    }
    
    private boolean checkLength(String password) { ... }
    private boolean containsUpperCase(String password) { ... }
    private boolean containsLowerCase(String password) { ... }
    private boolean containsDigit(String password) { ... }
    private boolean containsSpecialChar(String password) { ... }
}
```

从实践的角度来看:

+ 应该尽量避免巧合内聚和逻辑内聚,因为这们会导致代码难以维护和理解
+ 时间内聚在某些特定场景下是可以接受的,比如系统初始化
+ 过程内聚和通信内聚在业务逻辑中比较常见
+ 顺序内聚和功能内聚是我们应该追求的目标,它们能让代码更容易理解和维护

记住：内聚性越强,代码就越容易理解、测试和维护。在实际开发中,我们应该尽量往功能内聚的方向努力。

**哎呀，简单说就是你尽量要让一个类中的方法完成同一功能，遵循某个顺序，处理同一数据。**

### 耦合性
#### 概念
耦合性<font style="color:rgba(0, 0, 0, 0.87);">显示了模块之间的相互依赖性，表明软件结构中多个模块之间的相互连接。简单的连接性使得软件易于理解并减少“涟漪效果“的倾向。</font>

#### <font style="color:rgba(0, 0, 0, 0.87);">耦合的七个层次</font>
![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735095943452-dc06cdb1-0e46-47c9-a1e4-77822c602bdb.png)

1. **内容耦合：模块直接访问/修改另一个模块的内部数据**

```java
class ModuleA {
    private int data = 10;
}

class ModuleB {
    // 直接访问ModuleA的私有数据，这是内容耦合
    void method(ModuleA a) {
        int value = a.data; // 违反封装
    }
}
```

2. **公共耦合：多个模块共享同一个全局数据**

```java
class GlobalData {
    public static int sharedCounter = 0;
}

class ModuleA {
    void increment() {
        GlobalData.sharedCounter++; // 多个模块共享数据
    }
}

class ModuleB {
    void decrement() {
        GlobalData.sharedCounter--; 
    }
}
```

3. **外部耦合：多个模块依赖同一个外部格式**

```java
class ModuleA {
    void saveToFile(String data) {
        // 依赖特定的文件格式
        Files.write(Paths.get("data.csv"), data.getBytes());
    }
}

class ModuleB {
    String readFromFile() {
        // 同样依赖该文件格式
        return Files.readString(Paths.get("data.csv"));
    }
}
```

4. **控制耦合：一个模块控制另一个模块的执行流程**

```java
class Processor {
    void process(String type) {
        if(type.equals("A")) {
            processTypeA();
        } else {
            processTypeB();
        }
    }
}

class Client {
    void execute() {
        Processor p = new Processor();
        p.process("A"); // 控制Processor的执行流程
    }
}
```

5. **标记耦合：模块间传递的数据结构中包含无用信息**

```java
class ComplexData {
    private int id;
    private String name;
    private String address; // 对某些操作来说是多余的
}

class Service {
    void processName(ComplexData data) {
        // 只需要name，但传入了整个对象
        System.out.println(data.getName());
    }
}
```

6. **数据耦合：模块间只通过参数传递数据**

```java
class Calculator {
    int add(int a, int b) { // 良好的数据耦合
        return a + b;
    }
}

class Client {
    void calculate() {
        Calculator calc = new Calculator();
        int result = calc.add(5, 3);
    }
}
```

7. **非直接耦合：模块间没有直接关系，但修改一个可能影响另一个**

```java
class Database {
    void save(Data data) {
        // 保存数据
    }
}

class Cache {
    void invalidate() {
        // 清除缓存
    } // 数据库操作可能需要清除缓存，但两者没有直接关联
}
```

这些耦合类型按照耦合度从高到低排序：内容耦合 > 公共耦合 > 外部耦合 > 控制耦合 > 标记耦合 > 数据耦合 > 非直接耦合。在实际开发中，应该尽量降低耦合度，推荐使用数据耦合。

#### 总结
![画板](https://cdn.nlark.com/yuque/0/2024/jpeg/42768076/1735227189353-12b04714-82be-4400-a664-3491e9f0ac75.jpeg)

> 不同的类尽量不要依赖于外部或者其他类内部的变量，只依赖于自己类内部的变量，两个类之间依赖传参进行沟通，并且传参不要传递没用的参数（只有少部分情况依赖像数据库这样的外部进行沟通）。
>
> 类内部的方法也尽量依赖于自己自身的变量，而不是类的属性。
>
> 另外一种情况，工具类中为另一个类提供执行流程，这个是正常的，当类内部的方法如果要相互调用，我们可以设置private方法，或提取到工具类中。
>

**<font style="color:rgba(0, 0, 0, 0.87);">软件设计时，尽量做到高内聚，低耦合。</font>**

## 软件测试
### 冒烟测试
+ **定义**：冒烟测试是针对软件的核心功能进行快速验证，确保软件的基本功能正常运行，一般用于构建之后的初步测试。
+ **适用场景**：快速检查软件是否具备进一步测试的条件。

### 回归测试
+ **定义**：回归测试是针对软件修改后，重新测试受影响模块以及相关模块，以确保修改没有引入新的错误。
+ **适用场景**：改动代码后需要验证改动的正确性，并检查改动是否破坏其他功能。

### 集成测试
+ **定义**：集成测试是将不同模块组装在一起测试接口和交互，主要关注模块之间的协作关系。
+ **适用场景**：用于多个模块结合之后，验证模块之间是否能够正确协同工作。

### 确认测试
+ **定义**：确认测试是验证软件是否按照需求规格说明书的要求工作，通常是面向需求的验证。
+ **适用场景**：主要关注需求是否实现，而非改动是否带来影响。

## 类设计的基本原则
1. 开闭原则：模块应该对扩展开放，对修改封闭。  
对扩展开放：允许通过添加新功能的方式扩展模块的行为。  
对修改封闭：在不改变模块现有代码的情况下，能够实现新的需求或功能。  
换句话说，尽量通过新增代码来实现新功能，而非修改已有代码，从而提高系统的稳定性和可维护性。
2. 里式替换原则：Liskov替换原则（LSP）是面向对象设计的基本原则之一，指如果一个类型S是类型T的子类型，则S类型的对象可以替换T类型的对象，并且程序不会产生错误或异常。 
3. 依赖倒置原则：依赖于抽象，而非具体实现 ；   
具体来说是程序要依赖于抽象接口，不要依赖于具体实现。简单的说就是要求对抽象进行编程，不要对实现进行编程，这样就降低了客户与实现模块间的耦合。
4. 接口分离原则：单一职责的多个专用接口优于一个臃肿的通用接口。  
核心思想：客户端不应该被迫依赖它不需要的接口。  
适用场景：当多个客户端使用同一个接口时，应将接口分割成更小、更专用的接口，确保每个接口只包含客户端需要的功能，从而降低接口的复杂性，增强系统的灵活性和可维护性。

## 什么是重构
重构是一种软件开发过程中的行为，指在不改变软件外部行为的前提下，通过修改代码内部结构和组织方式来改善软件质量，包括提高代码可读性、可维护性、可扩展性等。

## 测试
1、按是否查看程序内部结构分为：

（1）黑盒测试（black-box testing）：只关心输入和输出的结果 

（2）白盒测试（white-box testing）：去研究里面的源代码和程序结构

2、按是否运行程序分为：

（1）静态测试（static testing）： 

对于代码测试，主要是测试代码是否符合相应的标准和规范。

对于界面测试，主要测试软件的实际界面与需求中的说明是否相符。

对于文档测试，主要测试用户手册和需求说明是否真正符合用户的实际需求。

（2）动态测试（dynamic testing），是指实际运行被测程序，输入相应的测试数据，检查输出结果和预期结果是否相符的过程

3、按阶段划分：

（1）单元测试（unit testing），是指对软件中的最小可测试单元进行检查和验证。

（2）集成测试（integration testing），是指将通过测试的单元模块组装成系统或子系统，再进行测试，重点测试不同模块的接口部门。集成测试就是用来检查各个单元模块结合到一起能否协同配合，正常运行。

（3）系统测试（system testing），指的是将整个软件系统看做一个整体进行测试，包括对功能、性能，以及软件所运行的软硬件环境进行测试。系统测试的主要依据是《系统需求规格说明书》文档。

（4）验收测试（acceptance testing），指的是在系统测试的后期，以用户测试为主，或有测试人员等质量保障人员共同参与的测试，它也是软件正式交给用户使用的最后一道工序。

验收测试又分为a测试和beta测试，其中a测试指的是由用户、 测试人员、开发人员等共同参与的内部测试，而beta测试指的是内测后的公测，即完全交给最终用户测试。

4、黑盒测试分为功能测试和性能测试：

（1）功能测试（function testing），是黑盒测试的一方面，它检查实际软件的功能是否符合用户的需求。

包括逻辑功能测试、界面测试、易用性测试、兼容性测试

易用性测试：是指从软件使用的合理性和方便性等角度对软件系统进行检查，来发现软件中不方便用户使用的地方。

兼容性测试：包括硬件兼容性测试和软件兼容性测试

（2）性能测试（performance testing）：

主要有时间性能和空间性能两种

时间性能：主要指软件的一个具体事务的响应时间

空间性能：主要指软件运行时所消耗的系统资源。

5、软件性能测试：

（1）一般性能测试：指的是让被测系统在正常的软硬件环境下运行，不向其施加任何压力的性能测试。

（2）稳定性测试也叫可靠性测试（reliability testing）：是指连续运行被测系统检查系统运行时的稳定程度。

（3）负载测试（load testing）：是指让被测系统在其能忍受的压力的极限范围之内连续运行，来测试系统的稳定性。

（4）压力测试（stress testing）：是指持续不断的给被测系统增加压力，直到将被测系统压垮为止，用来测试系统所能承受的最大压力。

6、其他测试类型：

（1）回归测试（regression testing）是指对软件的新的版本测试时，重复执行上一个版本测试时的用例。

（2）冒烟测试（smoke testing），是指在对一个新版本进行大规模的测试之前，先验证一下软件的基本功能是否实现，是否具备可测性。

（3）随机测试（random testing），是指测试中所有的输入数据都是随机生成的，其目的是模拟用户的真实操作，并发现一些边缘性的错误

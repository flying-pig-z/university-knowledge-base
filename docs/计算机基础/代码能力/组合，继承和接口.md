## 组合和继承
+ 组合和继承的概念
+ 如果两个类只关心输入输出，只是单纯的调用关系，那么就用组合，直接调用方法。

> 组合相比于直接在每个地方new更加简洁，也更节约内存开销。
>

+ 如果涉及到两个类的变量交互，那么双方都要提供方法给对方调用，这时候会偏向于使用继承。（比如模板设计模式）

## 从业务链看继承
+ 如果两个类需要互相操作对⽅的属性，那么⼀定需要提供⽅法给对⽅调⽤，这就是天然的耦合，我们就需要考虑他们是不是本来就是⼀个东⻄ 使⽤继承关系将他们绑定起来看作是⼀个对象。
+ 继承不仅仅是对父类功能的扩展，更是使用继承链在实现业务链。自顶向下看继承，继承链的顺序就是业务链的顺序。
+ 经典的Interface --- AbsClass --- SubClass 结构【接口 -- 抽象类 -- 具体类】
+ 职责拆分：[1]拆分成多个类便于维护  
[2为了以后更好的切面和拓展。比如Spring把一个Bean的处理流程拆分的很细，就是为了在Bean的各个流程就行控制。

1. 实线箭头（绿色线条）：代表类之间的继承(extends)关系 
    - 比如 HierarchicalBeanFactory、ConfigurableBeanFactory、ListableBeanFactory 都继承自 BeanFactory
2. 虚线箭头（绿色虚线）：代表接口实现(implements)关系 
    - 比如 DefaultListableBeanFactory 实现了 ConfigurableListableBeanFactory 接口
3. 实线蓝色箭头：代表类之间的依赖关系 
    - 比如 DefaultSingletonBeanRegistry 到 FactoryBeanRegistrySupport 的连接，表示前者依赖或使用后者

最顶层：BeanFactory

+ 定义了最基础的业务：获取Bean这个核心操作
+ 相当于业务流程的起点：客户端需要一个Bean

第二层的分支体现了不同的业务场景需求：

+ ListableBeanFactory：需要遍历/查找Bean
+ HierarchicalBeanFactory：需要处理父子容器的层级关系
+ AutowireCapableBeanFactory：需要自动装配

往下走到ConfigurableBeanFactory和ConfigurableListableBeanFactory

+ 增加了配置和管理的业务需求
+ 体现了从"使用Bean"到"管理Bean生命周期"的业务扩展

最终到DefaultListableBeanFactory

+ 整合了所有上层的业务需求
+ 实现了完整的Bean生命周期管理流程

这个继承链反映的业务流程是：

最基础的获取Bean需求

↓

扩展出查找/遍历/父子容器/自动装配等具体场景

↓

增加配置和管理的能力

↓

最终形成完整的Bean工厂实现

就像一个业务从简单的"我要一个商品"，逐渐扩展成"我要能查找商品"、"我要能管理商品"、"我要能配置商品参数"等完整的业务流程。每一层继承都不是简单的代码复用，而是业务需求的自然延伸和补充。

## 多态与接口
+ 为什么接口之间也要继承？因为下游接⼝的对象，也需要⽤到上游接⼝定义的的⽅法，就和⼦类⽤到⽗类的⽅法⼀样。
+ 接口继承两⼤作⽤：1 功能组合(⾃下⽽上 多继承) 2 标记(⾃上⽽下 被多继承)
+ 接口最核心的作用就是定义抽象功能。接口的核心逻辑就是组合，就是搭积木，实现接口就代表实现什么功能。之后通过类的实现进⾏功能的组合 以及之后的⾯向抽象编程 ⾯向接⼝编程。（就和AOP效果本质一样）
+ 利用SPI，符合依赖倒置原则（抽象的指向具体的），程序依赖于抽象接口，不要依赖于具体实现。对抽象进行编程，不要对实现进行编程，这样就降低了客户与实现模块间的耦合。实现可插拔。
+ 接口的多继承本质上是能力的拓展。接口的多继承本质上是不同父接口能力的组合。

## 封装与管理
+ Java中的类简单分为4类 实体类、配置类(这⾥的配置是⼴义的配置 还包含了注册与 映射关系)、实现类(聚合类)、管理类。  
这在SpringBoot⽬录体系中⾮常明显 分别代表着 Domain、mapper(数据库关系映射)、Service(Impl)、Controller。  
有⼀个稍微特殊⼀些的就是mapper映射类 它表示着java对象与数据库操作之间的关 系 我们称为注册类或是映射类。

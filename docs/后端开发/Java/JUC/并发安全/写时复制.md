写操作在修改的时候，读操作可以读到脏数据。

> 就比如监控场景，我们一般会将监控的数据存在本地缓存中，然后开始一个定时任务定时刷新本地缓存，达到监控的效果。
>
> 这时候如果我们存放比如异常任务监控到CopyOnWriteList，先读取一部分，然后被覆盖后读取另一部分，就会导致前后两部分并不一致，参数并发问题。
>

其中一个解决方案就是加锁，但是会导致读的性能就会降低，因为每次读都要获取锁。

CopyOnWrite写时复制的核心原理是：读操作共享一个资源，写操作创建副本写入，写完毕之后复制给原资源。

当多个调用者（线程、进程等）请求相同资源时，会共享同一份资源，而不是各自创建副本。只有当某个调用者需要修改资源时，系统才会创建一个资源副本，供该调用者专用，其他调用者仍然共享原来的资源。

CopyOnWrite 在多种场景中应用广泛：

+ 操作系统的进程创建（fork 操作）
+ 并发集合类（如 Java 中的 CopyOnWriteArrayList）

> + CopyOnWriteArrayList -- 修改时候会创建副本，在副本上面修改，再去写入原来的ArrayList
>

+ 文件系统和数据库的快照实现

> MVCC -- 修改时候创建副本修改
>
> RDB AOF -- fork线程复制数据，不会阻碍原来的线程
>

+ 虚拟机内存管理

需要注意的是，CopyOnWrite 策略在写操作频繁的场景下效率会降低，因为会频繁创建对象副本。

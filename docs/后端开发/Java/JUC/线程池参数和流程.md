## 创建线程池的方式
主要就两种，一种使用Executors工厂类创建，里面内置了各种类型的线程

+ **固定线程池**：就像固定数量的员工。线程数量固定，超出的任务会在队列中等待
+ **缓存线程池**：就像高峰期临时加的兼职服务员。线程数量不固定，可根据需求自动创建，空闲线程会在60秒后被回收
+ **单线程池**：只有一个厨师的小吃摊。只有一个工作线程，保证所有任务按照顺序执行
+ **定时线程池** = 按时送外卖的配送员。支持定时及周期性任务执行
+ **工作窃取池** = 忙完自己区域就去帮其他区域的服务员。每个线程都有自己的双端队列，当一个线程空闲时，可以从其他线程队列末尾"窃取"任务

> 但是阿里巴巴开发规范不允许使用上面那种方法。
>
> 因为这些线程池的都是使用有界队列，也就是workQueue无限大，如果线程数达到maximumPoolSize且workQueue已满才不会堆积才会执行拒绝策略，所以如果workQueue无限大任务会一直堆积，导致OOM。
>

**所以我们一般使用ThreadPoolExecutor并自定义参数配置。**

## 线程池参数
ThreadPoolExecutor创建线程池的参数有哪一些呢？

1. corePoolSize：核心线程数
2. maximumPoolSize：最大线程数
3. keepAliveTime：非核心线程的空闲存活时间
4. unit：存活时间的单位，比如秒、毫秒
5. workQueue：任务队列
6. threadFactory：线程工厂
7. handler：拒绝策略

## 线程池完整流程
这么多参数怎么记呢？只要理解透了线程池的完整的工作流程，理解并不难。

1. 提交任务：代码通过execute，submit和invokeAll，invokeAny等方法提交任务给线程池

> + 当任务通过execute()提交到线程池，执行中抛出异常没有捕获线程池会终止并销毁线程
> + 当任务通过submit()提交任务，发生异常线程会复用而不会销毁。  
异常会被封装在由submit()返回的Future对象中。当调用Future.get()方法时，可以捕获到一个ExecutionException
>

2. 检查核心线程：如果当前运行的线程数小于corePoolSize，则创建新线程处理任务，即使其他线程是空闲的
3. 任务入队：如果运行的线程数已达到corePoolSize，任务会被放入workQueue等待执行
4. 创建非核心线程: 如果workQueue已满，但运行的线程数少于maximumPoolSize，创建新的非核心线程来处理任务
5. 拒绝任务:** 如果线程数已达到maximumPoolSize且workQueue已满**，触发handler指定的拒绝策略

## 参数详解
### 1. corePoolSize (核心线程数)
线程池尝试维护的目标线程数，设置为处理正常负载的线程数。

即使线程空闲，核心线程也不会被回收（除非设置allowCoreThreadTimeOut为true）

### 2. maximumPoolSize (最大线程数)
线程池允许创建的最大线程数，仅当任务队列已满时才会创建超过核心线程数的线程，应考虑系统资源限制设置合理值。

### 3. keepAliveTime (非核心线程空闲存活时间)
非核心线程空闲超过这个时间会被回收，仅对超出核心线程数的线程生效。

如果设置allowCoreThreadTimeOut为true，则核心线程也受此参数控制

### 4. unit (时间单位)
指定keepAliveTime的时间单位。比如TimeUnit.SECONDS、TimeUnit.MILLISECONDS等。

### 5. workQueue (任务队列)
存储等待执行的任务

常见队列类型： 

+ **ArrayBlockingQueue**: 有界队列，基于数组
+ **LinkedBlockingQueue**: 可选有界/无界队列，基于链表
+ **SynchronousQueue**: 不存储任务，提交必须立即有线程处理（相当于没有队列）
+ **PriorityBlockingQueue**: 优先级队列，任务按优先级执行
+ **DelayQueue**: 延时队列，元素需要延迟指定时间才能被取出

### 6. threadFactory (线程工厂)
创建新线程的工厂。

可自定义线程名称、优先级、是否为守护线程等，用于管理线程资源和实现特定需求（如定制线程名称便于调试）。

### 7. handler (拒绝策略)
当线程池和队列都满时的处理策略

Java提供四种标准实现： 

+ **AbortPolicy**: 直接抛出RejectedExecutionException异常（默认）
+ **CallerRunsPolicy**: 在提交任务的线程中直接执行被拒绝的任务（最常用）

> CallerRunsPolicy可以保证任务不丢失，处理提交任务的线程是主线程且工作量大，可能会导致主线程阻塞
>

+ **DiscardPolicy**: 直接丢弃任务，不做任何处理
+ **DiscardOldestPolicy**: 丢弃队列中头部（等待最久）的任务，然后尝试重新提交到线程池

> 如果工作队列是优先级队列，会丢弃优先级最低的任务
>

## 实际设置参数
| 参数 | 公式 | 说明 |
| --- | --- | --- |
| 核心线程数 | 平均QPS × 平均任务数 × 平均RT | 处理日常负载 |
| 最大线程数 | 峰值QPS × 平均任务数 × P99 RT | 应对峰值，但不用最极端值 |


核心线程数用 Little's Law：平均QPS × 平均任务数 × 平均响应时间（为了频繁销毁线程还可以在这个理论值基础上*1.2）

最长响应时间往往是异常情况，所以这里用峰值QPS × 平均任务数 × P99响应时间，或者简单地取核心线程数的 2-3 倍，还要考虑机器本身核心线程数

队列容量

+ 不宜过大，否则任务堆积导致响应延迟
+ 建议：`最大线程数 × 单任务执行时间内可处理的任务数`

JVM(Java虚拟机)的类加载采用了双亲委派模型，这是一种类加载器的层次结构设计。

## 基本原理
双亲委派机制的核心原则是：**当一个类加载器收到类加载请求时，它首先将请求委派给父类加载器完成**。只有当父类加载器无法找到或加载类时，子类加载器才会尝试自己加载。

## 类加载器层次结构
JVM中的类加载器通常构成以下层次结构(自顶向下):

1. **启动类加载器(Bootstrap ClassLoader)**
+ 最顶层的加载器，由C++实现，是JVM的一部分
+ 负责加载Java核心类库(`<JAVA_HOME>/lib`目录下的类)
+ 如`java.lang.*`、`java.util.*`等基础包
2. **扩展类加载器(Extension ClassLoader)**
+ 由Java实现，继承自`URLClassLoader`
+ 负责加载扩展目录(`<JAVA_HOME>/lib/ext`)下的类库
+ JDK 9之后被平台类加载器(Platform ClassLoader)取代
3. **应用类加载器(Application ClassLoader)**
+ 也称为系统类加载器(System ClassLoader)
+ 负责加载应用程序类路径(classpath)上的类库
+ 是大多数应用程序默认的类加载器
4. **自定义类加载器(User-defined ClassLoader)**
+ 开发者可以创建自己的类加载器
+ 通常继承自`ClassLoader`或`URLClassLoader`

## 工作流程
当需要加载一个类时，双亲委派模型的工作流程如下：

1. 类加载请求到达一个类加载器

> 默认情况下进入应用类加载器，JVM核心类请求直接从启动类加载器开始，如果有自定义类加载器则进入自定义类加载器
>

2. 该加载器首先检查自己的缓存中是否已加载该类
3. 若未加载，则将请求委派给父加载器
4. 请求最终传递到启动类加载器
5. 启动类加载器检查自己是否能加载(在自己的搜索范围内找到类)
6. 若启动类加载器无法加载，则请求下传给扩展类加载器
7. 若扩展类加载器也无法加载，则请求下传给应用类加载器
8. 若应用类加载器也无法加载，则抛出`ClassNotFoundException`

## 双亲委派的优势
1. **保证Java核心API的安全性**

防止用户自定义的类替换Java核心类，即使用户定义同名类，也会优先加载JDK内部的类。

2. **避免类的重复加载**

父加载器已加载的类不会被子加载器再次加载，提高了效率并节省了内存空间。

3. **保证类的唯一性**

同一个类(相同的全限定名)在同一个类加载器下只会被加载一次，不同的类加载器可以加载同名的类，相互隔离。

## 如何打破双亲委派机制
在某些场景下，可能需要打破双亲委派机制：

1. **重写**`loadClass()`**方法**

默认实现了委派逻辑，可以重写此方法改变行为

2. **使用线程上下文类加载器(Thread Context ClassLoader)**

Java提供的一种特殊机制，允许高层代码调用底层实现，如JDBC、JNDI等SPI机制。

3. **使用**`findClass()`**而非重写**`loadClass()`

推荐的做法是重写`findClass()`而非`loadClass()`，这样可以保留委派机制的核心逻辑，只修改实际查找类的实现。

## 为什么不直接到达启动类加载器，然后往下搜索？
1. 性能

从下往上搜索可以立马走缓存返回，而如果直接到启动类加载器，会导致启动类加载器负担过重。

2. 灵活性和可扩展性

每个类加载器可以决定是否委派给父加载器，可以通过重写`loadClass()`方法来修改委派行为，允许中间层的加载器拦截和处理特定的类加载请求。

而如果直接在顶层，需要显示的把请求传递给哪个类加载器，而且需要在顶层实现判断逻辑。

### Java中常见的垃圾器
![](https://cdn.nlark.com/yuque/0/2025/jpeg/42768076/1743945147146-22dd535e-8fde-4d53-9d09-06417a1926e6.jpeg)

+ Serial (单人清扫)  -- STW
    - Serial -- 单线程年轻代，复制算法
    - Serial Old -- 单线程老年代，标记整理算法
+ Parallel (多人同时清扫，但需要清场)  -- STW
    - ParNew -- 多线程年轻代，复制算法
    - Parallel Old -- 多线程老年代，标记整理算法
+ CMS (边工作边清扫，减少打扰)  -- 老年代，标记清除算法，部分过程STW
+ G1 (聪明地选择区域清扫)  --  全区域，标记排序清除算法，部分过程STW
+ ZGC (几乎无感知的清扫)  -- 全区域，标记整理算法（无感知），极少STW

最常用的是G1（现代JVM默认）、CMS（以前很流行）和ZGC（低延迟场景）。

### CMS收集器 -- Concurrent Mark Sweep
> **C**oncurrent（并发）：关键词是"并发"，记住CMS主要工作在并发模式下
>
> **M**ark（标记）：标记垃圾对象
>
> **S**weep（清除）：直接清除垃圾对象（不整理）
>

CMS是一种以获取最短停顿时间为目标的收集器，适用于对响应时间有高要求的应用。

**主要特点：**

+ 核心目标是减少停顿 STW 时间，所以以并发方式标记可回收对象
+ 采用"标记-清除"算法
+ 主要用于老年代收集

**工作流程：**

1. 初始标记（STW，短暂停顿）：标记GC Roots能直接关联的对象
2. 并发标记： 从初始标记的对象出发，递归地标记所有可达对象  
3. 重新标记（STW，短暂停顿）： 修正并发标记期间因用户程序运行导致的标记变动  
4. 并发清除： 清除所有没有被标记的对象（即垃圾对象）  

> 为什么需要停顿？
>
> 避免标记不一致， 如果在垃圾收集器标记对象的同时，应用程序还在不断修改对象引用关系，会导致标记结果不准确，造成错误回收。
>

**缺点：**

+ 占用CPU资源
+ 产生浮动垃圾
+ 产生内存碎片

### G1收集器 -- Garbage-First
G1是面向服务端应用的垃圾收集器，兼顾吞吐量和停顿时间，可以替代CMS。

"垃圾优先"，永远优先处理价值最高的垃圾区域

**主要特点：**

+ 将内存分成多个区域，不再区分新生代和老年代的物理空间划分
+ 可预测的停顿时间模型

**工作流程：**

1. 初始标记(STW)：标记GC Roots能直接关联到的对象  
2. 并发标记： 从GC Roots开始对堆中对象进行可达性分析，找出存活对象  
3. 重新标记（STW）： 处理并发标记阶段中未处理的引用变化  
4. 筛选回收（STW）：对各个Region的回收价值和成本进行排序，根据期望的停顿时间制定回收计划，执行垃圾回收

> 筛选回收阶段是G1收集器最具特色的部分，体现了"Garbage-First"的核心思想：
>
> 1. **价值导向**: 不是盲目回收所有垃圾，而是先回收"价值最高"的区域
> 2. **预测模型**: 根据历史数据，精确预测每个Region的回收时间和空间收益
> 3. **时间预算**: 根据用户设定的期望停顿时间，选择能在预算时间内完成的Region集合
> 4. **"贪心"算法**: 在有限时间内尽可能多地回收垃圾
>
> 这种工作方式使G1能够在有限的停顿时间内获得最大的回收效益，平衡了吞吐量和停顿时间的需求，特别适合需要大内存且对停顿时间敏感的应用。
>

**优势：**

+ 可精确控制停顿时间
+ 分区域的内存布局，可并行、并发处理
+ 基于Region的内存分配回收，减少内存碎片
+ 通过优先级队列，优先回收价值最大的区域，提高资源利用率

### ZGC -- Z Garbage Collector
想象一下你有一个很大的玩具房间，里面放满了玩具。有些玩具你还在玩（这就是活动对象），有些已经不玩了（这就是垃圾对象）。随着时间推移，房间变得越来越乱，需要整理。这就是垃圾收集器的工作。

ZGC就像一个特别聪明的管家，他能在你玩玩具的同时悄悄地整理房间，几乎不打扰你。传统的垃圾收集器需要你停下来不玩，等它整理完才能继续，而ZGC几乎不需要你停下来。

ZGC如何工作？

**1.标记重要玩具**

首先，ZGC会迅速扫一眼，看看哪些是你正在使用的重要玩具（从GC Roots开始）。这个过程非常快，只需要停顿1-2毫秒。

**2.找出所有重要玩具**

接下来，ZGC会在你继续玩的同时，顺着这些重要玩具，找到所有你可能会用到的玩具（追踪对象引用）。它会给这些玩具贴上特殊的标签（染色指针标记）。

> 三色标记算法是一种追踪式垃圾收集算法，它将对象分为三种颜色：
>
> 1. **白色**：尚未被垃圾收集器访问过的对象。在标记开始时，所有对象都是白色的。最终，所有依然是白色的对象会被视为垃圾进行回收。
> 2. **灰色**：已经被垃圾收集器发现，但是其引用的对象还没有全部被扫描完的对象。灰色对象代表正在处理的"边界"。
> 3. **黑色**：已经被垃圾收集器访问过，且其所有引用的对象也都已经被访问过的对象。黑色对象被认为是"存活"的，不会被回收。
>
> 标记过程大致如下：
>
> 1. 初始时，所有对象都是白色
> 2. 将GC Roots直接引用的对象标记为灰色，放入"灰色集合"
> 3. 从灰色集合中取出一个对象： 
>     - 将其标记为黑色
>     - 将其引用的所有白色对象标记为灰色，并加入灰色集合
> 4. 重复步骤3，直到灰色集合为空
> 5. 此时，所有剩余的白色对象就是垃圾，可以被回收
>

**3.选择要整理的区域**

ZGC会选择一些特别乱的区域进行整理，主要是那些有很多垃圾玩具的地方。

**4.搬运玩具**

然后，ZGC开始把重要玩具搬到新的、整洁的区域。有趣的是，它不需要你停止玩耍。如果你突然想玩一个正在被搬运的玩具：

+ 如果玩具已经搬走了，ZGC会告诉你新位置在哪里
+ 如果玩具还没搬，ZGC会标记它，稍后再搬

这个过程使用了"读屏障"技术，就像一个小助手，每当你拿起玩具时，它会检查这个玩具是否需要或已经被搬运。

**5.更新玩具之间的关系**

有些玩具之间是有联系的（比如积木和积木盒）。当玩具被搬走后，ZGC会更新这些关系，确保你能找到所有相关的玩具。

**6.清理垃圾**

最后，那些没有标记的玩具（垃圾）所占的空间被回收，可以用来放新玩具。

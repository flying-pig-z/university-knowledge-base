## 基础结构
HashMap就是数据结构中哈希表的实现，用于存储键值对，核心就是根据键的哈希值映射到数组中对应的索引处进行存储。

JDK8之前是使用数组+链表的结构，JDK8之后是使用数组+红黑树的结构。

**存入元素的过程**

HashMap 通过 key 的 hashcode()方法得到的hashcode，然后经过扰动函数（即hash方法）处理过后得到 hash 值，然后通过 (n - 1) & hash 判断当前元素存放的位置（这里的 n 指的是数组的长度）。

如果当前位置存在元素的话，就调用equals()方法判断该元素与要存入的元素的 hash 值以及 key 是否相同，如果相同的话，直接覆盖，不相同就通过链表或者红黑树解决冲突。

**取出元素的过程**

首先通过 key 的 hashCode() 方法获取哈希值，然后通过扰动函数（hash 方法）处理获得 hash 值，通过 (n - 1) & hash 计算出数组下标位置（n 为数组长度）。

检查数组该位置的元素，如果为空，则返回 null；如果不为空，则进行下一步判断。判断第一个节点，如果第一个节点的 hash 值和 key 都与查找的元素相同，则直接返回该节点的 value，如果不相同，则继续查找 。

如果是红黑树节点，则按照红黑树的方式进行查找；如果是链表节点，则遍历链表查找。比较每个节点的 hash 值和 key 是否与要查找的元素相同，如果找到，返回该节点的 value，如果遍历完都没找到，返回 null。、

> 了解了插入和查取的完整过程，就不难理解为什么重写了equals也要重写hashcode方法。不然根据equals
>

**补充：扰动函数**

HashMap 中的扰动函数（hash 方法）是用来优化哈希值的分布。通过对原始的 hashCode() 进行额外处理，扰动函数可以减小由于糟糕的 hashCode() 实现导致的碰撞，从而提高数据的分布均匀性。

下面是JDK8扰动函数的实现：

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

## 处理哈希冲突
### 链表
JDK7之前是使用数组+链表来处理哈希冲突。

**将链表和数组相结合。也就是说创建一个链表数组，数组中每一格就是一个链表。若遇到哈希冲突，则将冲突的值加到链表中即可。**

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735450860603-c5d85d08-e017-443e-9b18-54d9cdd7571a.png)

在JDK1.7及之前链表的插入采用的是头插法，即每当发生哈希冲突时，新的节点总是插入到链表的头部，老节点依次向后移动，形成新的链表结构。

在多线程环境下进行扩容插入操作，头插法可能导致链表结构出现环路，从而引发死循环，最终导致程序卡死或无限循环。

### 红黑树优化
JDK8之后引入红黑树进行优化，变成数组+链表/红黑树。

当链表长度大于阈值（默认为 8）时，将链表转化为红黑树，以减少搜索时间。

将链表转换成红黑树前会判断，如果当前数组的长度小于 64，那么会选择先进行数组扩容，而不是转换为红黑树。

当链表长度降到6以下时，红黑树会重新退化为链表，保持简单高效。

红黑树是一种平衡二叉搜索树，插入、删除、查找操作的时间复杂度为 O(log n)，在元素多的情况下远优于链表的 O(n)。

**若遇到哈希冲突，则将冲突的值加到链表/红黑树中即可。**

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1735450869211-a0474c71-0858-49c5-af20-de8d520a1df0.png)

> TreeMap、TreeSet 以及 JDK1.8 之后的 HashMap 底层都用到了红黑树。红黑树就是为了解决二叉查找树的缺陷，因为二叉查找树在某些情况下会退化成一个线性结构。
>

<font style="color:rgb(60, 60, 67);">将链表转换成红黑树前会判断，如果当前数组的长度小于 64，那么会选择先进行数组扩容，而不是转换为红黑树。</font>

> + <font style="color:rgb(60, 60, 67);">为什么数组要大于等于64才转红黑树？</font>
>     - <font style="color:rgb(60, 60, 67);">主要原因有以下两点：避免频繁的树化以及减少内存占用。</font>
>     - <font style="color:rgb(60, 60, 67);">当数组容量较小的时候，哈希冲突大，但是扩容后的HashMap自然会减少哈希冲突。</font>**<font style="color:rgb(60, 60, 67);">如果在小数组上过早地进行链表转红黑树，可能会因为很快的扩容导致不必要的树化开销。刚树化了没多久就扩容了，冲突就少了，此时不就白白树化了？所以设计上设置了数组大小需要大于64才允许链表转为红黑树，防止频繁的树化。</font>**
>     - **<font style="color:rgb(60, 60, 67);">红黑树相比链表需要更多的内存，尤其是在节点较少的情况下，红黑树的额外指针和结构占用更大。</font>**<font style="color:rgb(60, 60, 67);">为了节省内存，HashMap选择只有在数组容量达到一定规模后才树化，防止红黑树在小规模数据中带来额外的内存负担。</font>
> + <font style="color:rgb(60, 60, 67);">为什么不抛弃链表，直接使用红黑树？</font>
>     - <font style="color:rgb(60, 60, 67);">因为红黑树节点的大小是普通节点大小的两倍，所以为了节省内存空间不会直接只用红黑树，只有当节点到达一定数量才会转成红黑树，这里定义的是8。8是空间和时间的平衡。</font>
> + <font style="color:rgb(60, 60, 67);">为什么定了6而不是小于等于8就变？</font>
>     - <font style="color:rgb(60, 60, 67);">是因为要留个缓冲余地，避免反复横跳。举个例子，一个节点反复添加，从8变成9，链表变红黑树，又删了，从9变成8，又从红黑树变链表，再添加，又从链表变红黑树？所以余一点，毕竟树化和反树化都是有开销的。</font>
>

## 扩容过程
HashMap的默认初始容量为16，负载因子为0.75。也就是说，当存储的元素数量超过16×0.75=12个时，HashMap会触发扩容操作，容量x2并重新分配元素位置。这种扩容是比较耗时的操作，频繁扩容会影响性能。

> 16和0.75这个组合是在性能和空间之间找到的平衡。
>
> 较高的负载因子（如1.0）会减少空间浪费，但增加了哈希冲突的概率，降低查询效率；较低的负载因子会导致空间利用率低，频繁扩容，但减少哈希冲突，查询效率高。经过大量实践，0.75被认为是大多数场景下比较合适的值，能够在时间和空间之间取得良好的平衡。所以设置了0.75。
>
> 如果已知HashMap的容量需求，建议提前设定合适的初始容量，以减少扩容带来的性能损耗。
>

JDK1.8改进了扩容时的元素迁移机制。在扩容过程中不再对每个元素重新计算哈希值，因为每次都是扩容两倍，所以是根据原数组长度的高位来判断元素是留在原位置，还是迁移到新数组中的新位置。这一改动减少了不必要的移动和计算，提升了扩容效率。

> 下面针对的是MySQL存储引擎InnoDB

## 崩溃恢复流程
+ undo log（回滚日志）：是 Innodb 存储引擎层生成的日志，实现了事务中的原子性，主要用于**事务回滚和 MVCC**
+ redo log（重做日志）：是 Innodb 存储引擎层生成的日志，实现了事务中的持久性，用于记录物理修改，用于崩溃恢复的事务重做。
+ binlog （归档日志）：是 Server 层生成的日志，记录执行的SQL，用于**数据备份和主从复制**

崩溃恢复的流程：

【1】主节点

binlog判断去向，如果binlog已经写入，那么就照样执行redo log，如果binlog未写入，那么就undo log重做日志。（具体后面两阶段提交会讲）

redo log重做，实际数据页可能还在Buffer Pool中未刷盘，崩溃时Buffer Pool数据丢失，需要通过redo log恢复。

undo log恢复，有些操作已经写入Buffer Pool中，但是事务还未提交，undo log进行回滚。

【2】从节点

采用binlog将数据同步到从节点。

## redo log与binlog的两阶段提交
### redo log 和 undo log 不一致问题
binlog是一个事务或者N个事务执行完刷盘，而redo log默认策略是事务提交刷盘，但是事务开始更新记录就在缓存中，有后台线程每隔一秒同步到page cache然后异步刷盘。

所以binlog是在事务提交完之后刷盘，而事务没有提交redo log也有可能刷盘。两者同步时间不同，就会造成两份日志不一致。

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1738668182161-b5279196-3e68-4149-9333-10495f277144.png)

### 解决方案 -- 两阶段提交
为了解决两份日志之间的逻辑一致问题，InnoDB 存储引擎使用两阶段提交方案。 

原理很简单，将 redo log 的写入拆成了两个步骤prepare和commit，这就是两阶段提交。

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1738678401304-318d75ac-b929-42fd-aa59-ab1f0b376ca2.png)

两阶段提交分为以下两个阶段：

第一阶段（准备阶段）：

将事务的修改信息记录到redo log中，但不立即提交，此时redo log处于prepare状态。

第二阶段（提交阶段）：

将事务的修改信息记录到binlog中，InnoDB存储引擎将刚才prepare状态的redo log改为commit状态，事务完成提交。

在崩溃恢复时：

+ 如果prepare阶段完成，但binlog未写入，处于prepare状态的事务会回滚
+ 如果binlog写完，redo log状态还没更新为commit，由于binlog已经写入，系统会将redo log设置为commit状态，redo log继续负责重做事务

# 概述
数据库的恢复简单说就是把数据库从错误状态恢复到某一已知的正确状态。

毕竟故障是难以避免的。

数据库的故障大概有以下几种：

+ 事务内部的故障
    - 恢复方法：撤销事务，UNDO日志
+ 系统故障（数据库崩了）
    - 恢复方法： 强行撤消（UNDO）所有未完成事务 重做（REDO）所有已提交的事务
+ 介质故障 -- 服务器残了，之前的数据库重启不了
    - 装入数据库发生介质故障前某个时刻的数据副本 + 重放Bin log

**其实这些恢复操作的基本原理就是冗余。利用存储在系统其它地方的冗余数据来重建数据库中已被破坏或不正确的那部分数据。**

恢复机制涉及到下面几个关键问题：

（1）如何建立冗余数据 

+ 数据转储（backup） 
+ 登录日志文件（logging） 

（2）如何利用这些冗余数据实施数据库恢复

# 数据转储
转储是指DBA将整个数据库复制到磁带或另一个磁盘上保存起来的过程，备用的数据称为后备副本或后援副本。

**简单说就是创建副本。**数据库遭到破坏后可以将后备副本重新装入， 重装后备副本只能将数据库恢复到转储时的状态。

## 静态转储与动态转储
### 静态转储
在系统中无运行事务时进行的转储操作，转储开始时数据库处于一致性状态，转储期间不允许对数据库的任何存取、修改活动，得到的一定是一个数据一致性的副本。

优点：实现简单

缺点：降低了数据库的可用性，转储必须等待正运行的用户事务结束，新的事务必须等转储结束

### 动态转储
转储操作与用户事务并发进行，转储期间允许对数据库进行存取或修改。

优点：

+ 不用等待正在运行的用户事务结束
+ 不会影响新事务的运行

缺点： 不能保证副本中的数据正确有效

[例]在转储期间的某个时刻Tc，系统把数据A=100转储到磁带上，而 在下一时刻Td，某一事务将A改为200。转储结束后，后备副本上 的A已是过时的数据了

那么如何利用动态转储得到的副本进行故障恢复呢？

需要把动态转储期间各事务对数据库的修改活动登记 下来，建立日志文件，后备副本加上日志文件才能把数据库恢复到某一时刻 的正确状态

## 海量转储与增量转储
海量转储: 每次转储全部数据库（更方便）

增量转储: 只转储上次转储后更新过的数据（效率高）

# 登记日志文件
## 概述
日志文件是用来记录事务对数据库的更新操作的文件。

MySQL的三大日志 -- Bin log, Redo log ,Undo log。

它们的作用：

Bin log -- 记录所有数据库修改操作的SQL语句，用于主从复制，备份和重装数据库。

Redo log --  记录对数据页修改前后的数据，比如"在第100页第1行改了个值，修改前是..，修改后是.."，主要用于数据库崩溃恢复和提升性能。  
【提升性能：先写日志，不用立即把数据写入磁盘】

Undo log --  记录数据被修改前的值，用于事务回滚和MVCC。

> 为什么Bin log和Redo log要分开记录呢？
>
> 1. Bin Log记录SQL，便于理解和复制，不同的数据库都能看懂
> 2. Redo Log执行快，恢复效率高，但只有特定引擎能理解
>

## 日志文件的格式和内容
### 二进制日志 Binlog 格式
+ Statement格式 
    - 记录的是SQL语句的原文
    - 优点：记录的日志量较小，节省存储空间
    - 缺点：某些语句可能在从库执行结果与主库不一致，如now()、uuid()等函数
+ Row格式 
    - 记录的是每行数据的变化情况
    - 优点：记录的是真实的数据变更，主从复制更可靠
    - 缺点：日志体积较大，占用更多存储空间
+ Mixed格式 
    - 结合了Statement和Row两种格式的优点
    - 默认使用Statement格式
    - 在某些情况下会自动切换为Row格式，如使用了UUID()函数

### Undo Log格式
下面是undo log记录的核心信息（注意只是核心信息）：

事务标识信息：

事务ID 回滚指针[指向该记录的上一个版本] 表ID：标识这条记录属于哪个表

还有位置信息：

表空间ID 页号记录 在页内的偏移量 受影响的索引信息等

**最重要是记录了数据操作相关，便于回滚，包含：**

+ **INSERT操作：记录新插入记录的主键信息，便于回滚时删除**
+ **DELETE操作：记录删除记录的整行信息，便于回滚时恢复**
+ **UPDATE操作：记录修改前的旧值，便于回滚时还原**

### Redo Log格式
```plain
redo log header: {
    LSN (Log Sequence Number)
    日志类型
    空间ID
    页号
    数据偏移量
    具体修改的数据
}
```

Redo Log记录的是物理层面的页面修改操作，主要包含：

+ 页号（表空间号 + 页号）
+ 偏移量（在页面中的偏移量）
+ 修改前数据
+ 修改后数据

# 结合日志文件数据库恢复
## 事务出错回滚
通过事务的 undo log 段进行回滚，按照版本链反向操作:

+ 对于 INSERT,删除插入的记录
+ 对于 DELETE,根据 undo log 中的记录恢复数据
+ 对于 UPDATE,根据 undo log 中的记录恢复原值

> 注：undo log 同时服务于事务回滚和 MVCC，每个修改操作都会产生一个新的版本记录，回滚时按版本链反向操作即可，而不需要向数据库理论课上所说的对日志文件进行反向扫描。
>

## 系统故障的恢复
**系统故障造成数据库不一致状态的原因：**

+ 未完成的事务对数据库的更新可能已写入数据库
+ 已提交的事务对数据库的更新还留在缓冲区没来得及写入数据库

**恢复方法：**

1.Undo故障发生时未完成的事务

2.Redo已完成的事务

注：系统故障的恢复由系统在重新启动时自动完成，不需要用户干预。

**InnoDB故障恢复具体过程:**

1. **REDO阶段** (前滚)

从最后一个检查点开始扫描 redo log，重做所有页面修改,使数据页恢复到崩溃前的状态，包括未提交事务的修改也会被恢复。

2. **UNDO阶段** (回滚)

通过 InnoDB 的事务系统识别崩溃时未完成的事务，利用 undo log 中的信息回滚这些事务，回滚是通过回滚段中的版本链进行的,按照记录的反向操作进行。

注:InnoDB 使用单独的 redo log 和 undo log，并且使用检查点机制(checkpoint)来缩小需要扫描的日志范围，后面有介绍。

## 介质故障的恢复
1.重装数据库

2. 应用bin log重放事务恢复到故障发生前的某个时间点  

具体恢复步骤：

1. 装入最新的备份(离故障发生时刻最近的转储副本) ， 使数据库恢复到最近一次转储时的一致性状态。 
2. 使用 Binary Log 重放事务到指定时间点
3. 使用 mysqlbinlog 工具处理 Binary Log
4. 可以指定具体的恢复时间点

> 理论课上并不是利用Bin log，而是利用redo+undo。
>
> 首先扫描日志文件，找出故障发生时已提交的事务的标识， 将其记入重做队列。
>
> 然后正向扫描日志文件，对重做队列中的所有事务进行重做处理。即将日志记录中“更新后的值”写入数据库。
>

为什么有Bin log还需要载入备份？

+ 仅依赖 binlog 恢复时需要重放所有历史操作，如果数据量大，重放时间会非常长，全量备份+binlog 增量的方式可以大大缩短恢复时间。
+ binlog本身可能存在问题 -- 文件损坏，记录不完整，日志断档，同步延迟。结合备份和Binlog可以增加恢复的可靠性。

# 检查点技术【简化版理论】
**注意：这里的检查点技术属于我们学校理论课上的内容。实际Mysql中Innodb引擎的实现比下面复杂，下面只是理论，但是大致思路是一样的。**

## 问题的提出
两个问题

+ 搜索整个日志将耗费大量的时间 
+ REDO处理：重新执行，浪费了大量时间

解决方案 -- 具有检查点的恢复技术

在日志文件中增加检查点记录，增加重新开始文件，恢复子系统在登录日志文件期间动态地维护日志。

## 具体内容
检查点记录的内容 

1. 建立检查点时刻所有正在执行的事务清单  
2. 这些事务最近一个日志记录的地址

重新开始文件的内容 -- 记录各个检查点记录在日志文件中的地址

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1734851960344-19cea9a7-0387-4e0a-bb18-db969bb4b0f8.png)

### 动态维护日志文件的方法 
周期性地执行如下操作：建立检查点，保存数据库状态。 

具体步骤是：   
1.将当前日志缓冲区中的所有日志记录写入磁盘的日志文件上   
2.在日志文件中写入一个检查点记录   
3.将当前数据缓冲区的所有数据记录写入磁盘的数据库中  
4.把检查点记录在日志文件中的地址写入一个重新开始文件

### 建立检查点
恢复子系统可以定期或不定期地建立检查点,保存数据库状态 

+ 定期 -- 按照预定的一个时间间隔，如每隔一小时建立一个检查点 
+ 不定期 -- 按照某种规则，如日志文件已写满一半建立一个检查点

## 利用检查点的恢复策略
使用检查点方法可以改善恢复效率 

**当事务T在一个检查点之前提交，对数据库所做的修改已写入数据库，写入时间是在这个检查点建立之前或在这个检查点建立之时，在进行恢复处理时，没有必要对事务T执行REDO操作。（重点）**

系统出现故障时，恢复子系统将根据事务的不同状态采取不同的恢复策略

![](https://cdn.nlark.com/yuque/0/2024/png/42768076/1734852424006-008713d4-5844-4f3f-b0ad-8dfcf2f6f759.png)

+ T1：在检查点之前提交 
+ T2：在检查点之前开始执行，在检查点之后故障点之前提交 
+ T3：在检查点之前开始执行，在故障点时还未完成 
+ T4：在检查点之后开始执行，在故障点之前提交 
+ T5：在检查点之后开始执行，在故障点时还未完成 

对应的恢复策略： 

+ T3和T5在故障发生时还未完成，所以予以撤销 
+ T2和T4在检查点之后才提交，它们对数据库所做的修改在故障 发生时可能还在缓冲区中，尚未写入数据库，所以要REDO 
+ T1在检查点之前已提交，所以不必执行REDO操作

## 利用检查点的恢复步骤
1.从重新开始文件中找到最后一个检查点记录在日志文件中的地址，由该地址在日志文件中找到最后一个检查点记录

2.由该检查点记录得到检查点建立时刻起所有正在执行的事务清单ACTIVE-LIST 

建立两个事务队列：  
UNDO-LIST	REDO-LIST 

把ACTIVE-LIST暂时放入UNDO-LIST队列，REDO 队列暂为空。

3.从检查点开始正向扫描日志文件，直到日志文件结束 

+ 如有新开始的事务Ti，把Ti暂时放入UNDO-LIST队列 
+ 如有提交的事务Tj，把Tj从UNDO-LIST队列移到REDO- LIST队列 

4.对UNDO-LIST中的每个事务执行UNDO操作 

对REDO-LIST中的每个事务执行REDO操作

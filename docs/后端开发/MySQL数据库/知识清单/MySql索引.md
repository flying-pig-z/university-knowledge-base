MySql InnoDB的索引分为主键索引（也叫聚簇索引）与二级索引（也叫非聚簇索引）。

不管是主键索引还是二级索引，InnoDB都使用B+树结构。

## 主键索引（聚簇索引）
一个表只能有一个主键索引。

**主键索引的非叶子节点存储主键值和指针[页号]。**

> B+树的每个节点都存储在磁盘上的一个页中，一个页通常包含多个键值对[非叶子节点]或多条记录[叶子节点]。
>
> 每个页都有一个唯一的页号，用于标识该页的物理位置。非叶子节点中存储页号来指向其子节点所在页的地址（当成指向下一个结点的指针就好啦）。
>

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1742267150098-565baecf-4247-4398-be98-2f18e964e6a5.png)

**而叶子节点则包含了完整的数据[不止一条,有一页,16KB]。**

> B+树只能帮忙快速定位到页，而不是记录。
>
> 每个结点内部还有页目录，内部记录是单链表连接，通过页目录二分定位到对应的分组再遍历分组内的链表就可以得到对应的记录。
>
> 为什么要二分完再遍历，而不能直接二分？
>
> 因为页目录中的槽并不是指向页中的每一条记录，而是指向对应分组的最大记录。每个分组内都会包含若干条记录。
>

**叶子节点之间有双向链表连接，便于范围查询。**

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1742267136404-b76e9397-08b7-4da2-b5aa-4b3675e465f9.png)

> 实际上，每个分组的记录数是有规定的，图中做了省略只画了两条，InnoDB规定：
>
> + 第一个分组只有一条记录
> + 中间的分组4-8条记录
> + 最后一个分组1-8条记录
>
> 因此不必担心遍历很长的链表导致性能问题。
>



**查询过程：从根节点开始， 通过比较查询数据与节点内数据来确定区间，顺着指针进入下一层节点，重复这个过程直到叶子节点，找到该主键对应数据所在的数据页中。**

**然后在叶节点中通过页目录二分查找到对应的分组，再遍历分组中的数据链表得到数据页中对应的记录。**

> 以上图为例，比如说要查找35，就先从根结点入手，发现35在22-60之间，就走到第二层的第二个结点，然后发现35在30-65之间，就走这个结点的第二个子结点，然后在这个子结点中二分查找目录找到对应的分组，在遍历分组中的链表找到对应的记录。
>
> 注：每个结点都是目录+链表的形式，这里非叶子节点确立区间和叶子节点一样，也是先二分目录到对应分组，再遍历分组中的链表。
>

**像这种索引结构和数据一起存放的索引，也称为聚簇索引。InnoDB中主键索引就是聚簇索引。**

## 二级索引（辅助索引）
除了主键索引，其他什么普通索引 唯一索引都是二级索引。一个表可以有多个二级索引。

二级索引的非叶子节点：存储索引列值和指向下层的指针[页号]，而二级索引的叶子节点不存储完整数据行，而是存储索引列值和主键值。

如图

```plain
               [age:20]                // 非叶子节点：存储索引列值和指针
              /        \              
             /          \    
        [age:10]     [age:30]         // 非叶子节点
        /     \       /     \
   [age:5,  [age:10, [age:25, [age:30,  // 叶子节点：存储索引列值和主键值
    id:1]    id:2]    id:7]    id:8]
   [age:8,  [age:15,           [age:35,
    id:3]    id:4]              id:9]
            [age:18,
             id:5]
```

> 这里是简化版模型，只展示两路，实际是多路B+树。
>

覆盖索引：查询的字段正好是索引的字段或者一部分（一个索引包含所有需要查询的字段的值）



查询过程：从二级索引的根节点开始查找，查找过程和主键索引一样。  
如果覆盖索引，就可以在非叶子节点和叶子节点中获取该数据。  
如果不覆盖索引，就需要查找要在叶子节点获取主键值，使用获得的主键值到主键索引中查找完整数据（称为回表查询）。



像这种索引结构和数据分开存放的索引，也称为非聚簇索引。

> InnoDB中二级索引都是非聚簇索引，而MyISAM中不管是主键还是非主键，都是非聚簇索引。
>

## 联合索引
使用表中的多个字段创建的单个索引。

好处：

+ 容易覆盖索引 
+ 适用于多条件查询 
+ 减少索引数量 
+ 便于索引下推

> select age,city from user where age = '111' and city = '666' 
>
> age和city都是索引，但不是联合索引，这种情况需要回表，但是如果是联合索引就不需要回表。
>

## 范围查询与B+树
B+树与B树的区别就是B+树会将 数据/指向聚簇索引的主键 存放在叶子结点，而叶子结点只存放索引值。

这样就可以方便范围查询。

例如：范围查询 18 <= key <= 28 的完整流程。

+ 找到第一个等于最小值18的叶子节点
+ 在叶子节点层进行横向遍历，知道超过了28停止遍历
+ 将收集到的所有数据返回给用户

## 最左前缀匹配原则
#### 最左前缀匹配原则
**<font style="color:rgb(60, 60, 67);">在使用联合索引时，MySQL 会根据索引中的字段顺序，从左到右依次匹配查询条件中的字段。如果查询条件与索引中的最左侧字段相匹配，那么 MySQL 就会使用索引来过滤数据，这样可以提高查询效率。</font>**

**<font style="color:rgb(60, 60, 67);">最左匹配原则会一直向右匹配，直到遇到范围查询（如 >、<）为止。对于 >=、<=、BETWEEN 以及前缀匹配 LIKE 的范围查询，不会停止匹配。</font>**

#### <font style="color:rgb(60, 60, 67);">原理</font>
****我来结合B+树索引结构详细解释最左前缀匹配原则的工作原理。

（1）联合索引的物理结构：

假设我们有一个联合索引 (name, age, city)：

```plain
非叶子节点结构：
[name1-age1-city1]     [name2-age2-city2]
    ↙         ↘           ↙         ↘

叶子节点结构：
[name, age, city, 主键值] -> [name, age, city, 主键值] -> ...
```

数据在B+树中是按照索引列的顺序进行排序的：

首先按name排序，name相同时按age排序 ，name和age都相同时按city排序。例如：

```plain
非叶子节点：
[张三,20,北京]      [李四,25,上海]      // 存储索引值和指针
    /                   |              \    
    
叶子节点：
[张三,18,深圳,id:1] -> [张三,20,北京,id:2] -> [张三,22,上海,id:3] -> [李四,25,上海,id:4]...
```

（2）匹配原则分析

场景一：完全匹配

```sql
SELECT * FROM table WHERE name='张三' AND age=20 AND city='北京';
```

这种场景索引使用效率最高。首先在B+树中定位name='张三'的节点，在相同name下继续匹配age=20，最后在相同name和age下匹配city='北京'。

场景二：匹配左前缀

```sql
SELECT * FROM table WHERE name='张三' AND age=20;
```

这种场景可以有效使用索引。先定位name='张三'的节点，然后在相同name下匹配age=20，无需使用city列。

场景三：跳过中间列

```sql
SELECT * FROM table WHERE name='张三' AND city='北京';
```

这种场景只能用上name列的索引。可以通过索引找到name='张三'的记录，由于age列被跳过，无法利用city列的索引，必须在匹配到name='张三'的记录后，逐行判断city='北京'。

场景四：只查询后面的列

```sql
SELECT * FROM table WHERE age=20 AND city='北京';
```

这种场景完全无法使用索引。因为B+树是先按name排序的，没有name值无法在树中定位起始位置，必须全表扫描。

#### 优化建议
1. 调整索引列顺序：

```sql
-- 如果经常需要查询name和city，很少用到age
CREATE INDEX idx_name_city_age ON table(name, city, age);
```

2. 创建合适的覆盖索引：

```sql
-- 如果经常需要单独查询age和city
CREATE INDEX idx_age ON table(age);
CREATE INDEX idx_city ON table(city);
```

3. 根据查询拆分索引：

```sql
-- 如果查询条件经常变化，可以创建多个针对性索引
CREATE INDEX idx_name_age ON table(name, age);
CREATE INDEX idx_name_city ON table(name, city);
```

## 索引下推（ICP）
**索引下推是MySQL 5.6引入的一种查询优化技术。**

**在不使用ICP的情况下，MySQL会先通过索引找到满足条件的主键值，然后回表查询完整记录，再对记录进行条件过滤。**

**而使用ICP后，MySQL会在索引遍历过程中，先过滤掉不满足条件的记录，减少回表次数。**

 举个例子：  

```sql
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(20),
    age INT,
    city VARCHAR(20),
    INDEX idx_name_city(name, city)
);

SELECT * FROM user WHERE name LIKE '张%' AND city = '北京';
```

不使用索引下推的执行流程：

1. 通过name LIKE '张%'在索引中找到所有姓张的记录的主键值
2. 通过得到的主键值回表查询完整记录
3. 最后过滤city = '北京'的条件

使用索引下推的执行流程：

1. 在索引中找到满足name LIKE '张%'的记录
2. **直接在索引中过滤city = '北京'的条件（因为idx_name_city索引包含了name列和city列）**
3. 只对过滤后的数据进行回表查询

性能差异： 假设满足"张姓"的记录有100条，其中"北京"的只有10条。不使用ICP需要回表100次，而使用ICP只需要回表10次。

**索引下推针对联合索引，核心是将WHERE条件的判断下推到索引层面，从而减少回表次数。**

> 注：前面联合索引的分析也蕴含了索引下推。
>

## 怎么建立索引
+ 选择不含NULL，频繁作为查询/排序/连接条件和查询结果，不会频繁更新的字段建立索引
+ 单表索引尽量不超过五个
+ 尽可能建立联合索引而不是单列索引，选择性高/大概率出现的放在前面，范围查询那个值要放在索引最后
+ 注意避免冗余索引
+ 字符串类型的字段使用前缀索引代替普通索引
+ 避免索引失效

**索引失效的情况**

【1】记忆口诀： 最左计算模糊或，范围转换需留神  

【2】具体解释：

+ **最左** - 指组合索引的最左匹配原则，比如有索引(a,b,c)，查询时必须从左往右使用，不能跳过中间的列。
+ **计算** - 指在索引列上的计算操作，包括数学运算、函数调用。例如：WHERE age + 1 = 20。
+ **模糊** - 指以%开头的模糊查询，比如LIKE '%abc' 这种左模糊查询或者LIKE '%abc%'这种全模糊查询。而 LIKE 'abc%' 这种右模糊是可以用到索引的。

> `<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"apple"</font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);"> </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"application"</font>``<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"banana"</font>``<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"berry"</font>``<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"cherry"</font>`
>
> <font style="color:rgb(15, 17, 21);">索引条目是按照 </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">a->b->c</font>`<font style="color:rgb(15, 17, 21);"> 的顺序排列的。所有以 </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"a"</font>`<font style="color:rgb(15, 17, 21);"> 开头的单词都排在一起，然后是所有以 </font>`<font style="color:rgb(15, 17, 21);background-color:rgb(235, 238, 242);">"b"</font>`<font style="color:rgb(15, 17, 21);"> 开头的，以此类推，所以如果是由模糊是能利用索引的。</font>
>

+ **或** - 指 OR 连接的条件，只要 OR 任一边的条件没有索引，整个查询就都用不上索引。解决方案：可以改写为 UNION。
+ **列表长度范围** - IN和NOT IN如果太多了，可能会开启全表扫描
+ **转换** - 指隐式类型转换，比如字符串和数字的自动转换，WHERE id = '1' (id 是整数类型)。

特别说明一下 `SELECT *`：它本身不会导致索引失效，但会造成下面的问题：

（1）无法利用索引覆盖带来的优化

（2）增加网络传输开销

（3）增加数据处理量



一个表的索引不能太多，然后索引要尽量覆盖所有查询场景，不能失效。

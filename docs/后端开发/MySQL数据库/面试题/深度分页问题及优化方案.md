## 深度分页问题
在MySQL中一张表有上千万的记录，查询使用LIMIT 0,10的速度非常快，而查询LIMIT 8000000,10的速度却很慢。

MySQL需要扫描m条记录并跳过这个偏移量m，才能将想要的n条记录返回。也就是说，m值越大，扫描的记录就会越多。我们的实验中使用了800万的偏移量，延迟必然增加。

## 优化方案
### 优化方案一：利用索引特性
MySQL的InnoDB引擎中，索引是一棵B+树。主键索引的非叶子节点存储的是主键和指向下一个节点的指针，而叶子节点存储的是具体的数据。

如果我们要查询id等于10的用户信息，可以利用索引快速查找到具体节点，避免全表扫描，大大减少扫描的行数，降低查询延迟。

优化SQL示例：

```sql
SELECT * FROM user WHERE id > 8000000 LIMIT 10
```

执行结果：几乎察觉不到延迟，结果立即显示。

这里充分利用了B+树的特性，快速定位到id为8000000的记录，然后继续扫描叶子节点后面的10条记录即可。

>  优化方案一的核心思想就是使用范围查询（WHERE id > 某值）来代替LIMIT的第一个参数（偏移量）  。
>

### 优化方案二：子查询加覆盖索引
当我们需要按时间排序并分页时，情况会更复杂。

先创建create_time的倒序索引：

```sql
-- 建立create_time索引
CREATE INDEX idx_create_time ON user(create_time DESC)
```

未优化的SQL：

```sql
SELECT * FROM user ORDER BY create_time DESC LIMIT 8000000,10
```

执行结果：耗时超过40秒，延迟非常明显。

虽然给create_time建立了索引，但查询的是所有字段（SELECT *）。这些字段不会存储在非主键索引上，而是在主键索引的叶子节点上。非主键索引存储的是主键id，所以在扫描到这些数据时需要回表操作，即拿着id去主键索引上查询具体数据来组装结果，每次回表都会增加查询延迟。

优化SQL：使用子查询避免回表操作

```sql
SELECT u.* FROM user u
JOIN (
    SELECT id FROM user ORDER BY create_time DESC LIMIT 8000000,10
) t ON u.id = t.id
```

执行结果：耗时0.99秒，速度显著提升。

这里子查询中只查询了id，在非主键索引create_time这棵树上避免了大量回表操作，最终只返回10个id进行JOIN操作才会回表查询，从而降低了查询延迟。

注意：如果条件不仅是create_time排序，还有其他字段，此时必须创建联合索引的B+树以避免回表操作。这种技术称为索引覆盖，即查询字段和条件不会导致回表查询主键索引，在非主键索引就可以完成查询和过滤。

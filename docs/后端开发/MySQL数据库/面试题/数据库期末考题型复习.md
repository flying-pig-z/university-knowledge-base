## 关系代数运算
### 并运算（R ∪ S）
将两个查询的结果合并，同时自动去除重复的行。

```sql
SELECT * FROM R
UNION
SELECT * FROM S;
```

### 差运算（R - S）  
 在第一个查询结果中去除第二个查询结果中也存在的行。

```sql
SELECT * FROM R
EXCEPT
SELECT * FROM S;
```

### 笛卡尔积（R × T）
 返回两个表中所有元组的组合。

```sql
SELECT * 
FROM R, T;
```

###  自然连接（R ⨝ T）  
按两个关系中相同的列名连接（`NATURAL JOIN`）或 通过 `ON` 指定条件（`INNER JOIN`）。

> 前面一种是对应自然连接的时候不带条件，后面一种是则是通用的，对应自然连接，不管携带或不携带条件情况都可以。
>

```sql
SELECT * 
FROM R NATURAL JOIN T;
```

```sql
SELECT * 
FROM R
INNER JOIN T
ON R.common_attribute = T.common_attribute;
```

### 投影（ΠA(R)）
 从表中选择特定列。  

```sql
SELECT A FROM R;
```

###  选择（σ条件(R)）  
从表中筛选满足特定条件的行。  

```sql
SELECT * FROM R WHERE 条件;
```

###  除法（R ÷ S）  
查询那些在关系 `R` 中满足与 `S` 的所有值匹配的属性，查询结果不含S的属性。

在SQL中没有直接的关键字，但可以通过嵌套查询实现。  

假设 `R(A, B)` ÷ `S(B)`：  

```sql
SELECT A
FROM R
WHERE B NOT IN (
    SELECT B
    FROM S
    WHERE B NOT IN (
        SELECT B
        FROM R
    )
);
```

> 多练，没什么其他办法，把关系代数连得和SQL一样就成功了。
>

## 关系理论
### **数据依赖**
#### 函数依赖
在关系模式 R(U,F) 中，设 A 和 B 是属性或属性组，若对于 R 中的任意两个元组 t1 和 t2，只要 t1[A]=t2[A]，就有 t1[B]=t2[B]，则称：

+ **A 函数确定 B**，或
+ **B 函数依赖于 A**，记为： A→B

**简单理解**：

+ 如果知道了属性 A 的值，就可以唯一确定属性 B 的值。
+ A 是决定属性，B 是被决定属性。

#### 完全函数依赖
+ 如果 A→B，且在 A 中移除任何属性后，A→B 不再成立，则称为**完全函数依赖**。
+ **例子**：对于关系 R(学号,课程号,分数)，分数完全依赖于 学号+课程号，记为： 分数(学号,课程号)→分数 
+ 如果只依赖学号或课程号中的任何一个，分数是无法确定的。

#### **部分函数依赖**
+ 如果 A→B，但在 A 中移除某些属性后，A→B 依然成立，则称为**部分函数依赖**。
+ **例子**：对于关系 R(学号,课程号,分数)，如果 "课程号" 就可以确定分数（不依赖学号），则存在部分函数依赖： 课程号→分数。

#### **传递函数依赖**
+ 如果 A→B 且 B→C，则存在一种**传递函数依赖**，记为： A→C
+ **例子**：对于关系 R(学号,班级号,辅导员)：
    - 学号 → 班级号
    - 班级号 → 辅导员
    - 因此，存在传递函数依赖：学号 → 辅导员。

#### 多值依赖（非函数依赖的一种扩展）
多值依赖是一种特殊的依赖关系（如：A→→B），描述 A 决定 B 的多个值组合，但 B 和其他属性相互独立。  
（这是 4NF 处理的问题，与函数依赖区分开来。）

#### 平凡函数依赖
平凡函数依赖是指一个属性集的子集（或者它自己）能够唯一地依赖于这个属性集本身。

**换句话说**：

+ 如果 X→Y 中，Y 是 X 的一部分（Y⊆X），那么这种函数依赖就是**平凡的**。
+ 它**不增加额外的信息**，因为这是显而易见的。

**举个例子**： 在关系 R(A,B,C) 中：

+ {A,B}→A 是平凡函数依赖，因为 A 是 {A,B} 的子集。
+ {A,B}→{A,B} 也是平凡函数依赖，因为它本质上就是“依赖于自己”。

#### 非平凡函数依赖
非平凡函数依赖是指一个属性集可以唯一决定另一个**不属于它自己**的属性集。

**换句话说**：

+ 如果 X→Y 中，Y**不完全包含在 X 中**（Y⊈X），那么这种函数依赖就是**非平凡的**。
+ 它揭示了真正的依赖关系，也就是“新信息”。

**举个例子**： 在关系 R(A,B,C) 中：

+ A→B 是非平凡函数依赖，因为 B 不属于 A。
+ A→C 也是非平凡函数依赖，因为 C 也不属于 A。

### 码和属性
#### 码
码是一个**候选键**，指的是在关系模式中，能够**唯一标识每一行元组**的最小属性集。

具有以下特性：

+ **唯一性**：码的值能够唯一标识每一个元组。
+ **最小性**：码是最小的属性集，即如果移除码中的任何一个属性，唯一性就无法保证。

例如，在关系模式 R(A,B,C) 中，如果 {A,B} 能唯一标识一个元组且没有冗余属性，则 {A,B} 是一个码。

#### 主属性（Prime Attribute）
**主属性**是指**属于某一个候选码的属性**。也就是说，关系模式中的某个属性如果出现在候选码中，它就是主属性。如果关系模式中有多个候选码，则主属性是所有候选码中包含的属性的集合。

例如：

+ 候选码为 {A,B} 和 {C}，则 A,B,C 都是主属性。

#### 非主属性（Non-prime Attribute）
+ 非主属性是指**不属于任何候选码的属性**。
+ 它们是关系模式中的普通属性，不具备唯一标识元组的能力。

例如：在候选码为 {A,B} 和 {C} 的关系中，如果属性 D 不属于任何候选码，则 D 是非主属性。

#### 全码
整个属性组是码，称为全码（All-key）

#### 外码
在R（U1）中，S（U2）中，若K属于U1，K属于U2，K是S的主码，且K不是R的主码，则称K为R的外码。

> **更准确的表达**
>
> **3NF --消除主属性对不包含它的候选关键字的部分和传递函数依赖--> 4NF**
>

通过具体的例子解释各个范式：

**1NF（第一范式）示例：**  
假设有一个学生选课表：

```plain
学号 | 姓名 | 所选课程
001 | 张三 | 数学,物理,化学
```

这违反了1NF，因为"所选课程"列包含多个值。修改为符合1NF：

```plain
学号 | 姓名 | 课程
001 | 张三 | 数学
001 | 张三 | 物理
001 | 张三 | 化学
```

**2NF（第二范式）示例：**  
假设有一个订单明细表：

```plain
订单号 | 产品编号 | 产品名称 | 客户编号 | 客户地址 | 订购数量
```

这违反了2NF，因为产品名称只依赖于产品编号，客户地址只依赖于客户编号。修改为符合2NF：

```plain
订单明细表：
订单号 | 产品编号 | 客户编号 | 订购数量

产品表：
产品编号 | 产品名称

客户表：
客户编号 | 客户地址
```

**3NF（第三范式）示例：**  
假设有一个学生成绩表：

```plain
学号 | 姓名 | 课程 | 分数 | 课程学分
```

这违反了3NF，因为课程学分只依赖于课程，而不依赖于学号。修改为符合3NF：

```plain
学生成绩表：
学号 | 姓名 | 课程 | 分数

课程表：
课程 | 课程学分
```

**BCNF（Boyce-Codd范式）示例：**  
假设有一个教师课程表：

```plain
教师 | 课程 | 教师所属院系
```

这违反了BCNF，因为存在这样的依赖关系：教师→院系，院系→课程。修改为符合BCNF：

```plain
教师表：
教师 | 所属院系

院系课程表：
院系 | 课程
```

**4NF（第四范式）示例：**  
假设有一个学生兴趣和社团表：

```plain
学号 | 兴趣爱好 | 参加社团
```

这违反了4NF，因为兴趣爱好和参加社团是相互独立的多值依赖。修改为符合4NF：

```plain
学生兴趣表：
学号 | 兴趣爱好

学生社团表：
学号 | 参加社团
```

每个范式都是建立在前一个范式的基础上，目的是减少数据冗余和提高数据一致性。在实际应用中，通常将数据库设计到3NF就足够了，因为更高的范式可能会带来过多的表连接操作，影响查询性能。

## 并发
**冲突可串行化**是一种事务调度的正确性标准，指的是一个事务调度（Schedule）可以通过交换不冲突的操作，将其调整为某个**串行调度**（Serial Schedule）。  
串行调度是指事务的所有操作严格按照某种顺序依次执行（即一个事务完全执行完后，另一个事务才开始执行）。

---

判断是否是冲突可串行化的步骤可以通过 **冲突图（或优先图，Precedence Graph）** 来解决。以下是具体步骤：

### 步骤一：分析冲突操作
冲突操作指的是以下几种情况：

1. **读-写冲突（RW）**：一个事务读取某个数据，另一个事务随后写入相同数据。
2. **写-读冲突（WR）**：一个事务写入某个数据，另一个事务随后读取相同数据。
3. **写-写冲突（WW）**：两个事务写入相同数据。

这些冲突需要确定事务之间的依赖关系。

---

### 步骤二：构建冲突图
1. **节点表示事务**：每个事务用一个节点表示（如 T1, T2, T3, T4）。
2. **边表示依赖关系**：如果事务 Ti 的某个操作必须在事务 Tj 的某个操作之前完成（存在冲突），则从 Ti 到 Tj 画一条有向边。

---

### 步骤三：检查是否存在环
1. 如果冲突图中不存在环，则调度是**冲突可串行化的**。
2. 如果冲突图中存在环，则调度**不是冲突可串行化的**。

前面主要讲了两阶段提交和三阶段提交。由于两阶段和三阶段主要是追求分布式事务的强一致性，我们都知道分布式系统中，由于服务是独立部署，且事务是无法跨服务进行统一管理，所以实现一致性是非常困难的。

## 最终一致性概念
为了解决这个问题，出现了最终一致性。最终一致性其实是由BASE理论发展而来，也就是说如果数据在一段时间内没有被另外的操作所更改，那么它最终会达到与强一致性过程相同的结果。

实现最终一致性的方案有很多，比如：

+ 本地消息表
+ 消息系统
+ TCC
+ SAGA等方式

今天我们主要来看本地消息表。

## 本地消息表方案
本地消息表是一种常用的设计模式，用于实现分布式事务的最终一致性。通过消息的生成、发送和业务逻辑的执行放到同一个本地事务中，并结合消息队列和重试机制，本地消息表可以保证消息可靠的发送和下游服务正确的消费。

本地消息表最早是由eBay提出的，下面是具体的实现流程。

### 实现流程
1. 本地事务阶段
    - 执行业务逻辑和记录消息数据在同一个本地事务内执行
    - 保证执行业务逻辑和记录消息数据的原子性
    - 要么都成功，要么都失败
2. 消息记录
    - 执行业务操作（如会员注册）
    - 记录业务数据
    - 记录消息数据（存为待发送状态）到消息数据表
3. 消息发送
    - 定时任务轮询待发送状态的消息
    - 将待发送的消息发送到MQ（可以是RocketMQ、RabbitMQ、Kafka等）
    - MQ返回ACK给定时任务
    - 定时任务更新消息状态为已发送，防止消息重复发送
4. 消息消费
    - 下游服务订阅MQ
    - 消费者接收消息并执行业务逻辑（如赠送新会员积分）
    - 业务逻辑执行成功后返回消费成功ACK给MQ

### 方案优点
1. 最终一致性保障
    - 通过定时任务轮询和重试机制
    - 确保分布式事务达到最终一致性
2. 系统解耦
    - 消息队列实现了中间的松耦合
    - 服务A和服务B可以无需直接调用
3. 高可用性
    - 即使消息暂时无法发送或消费者宕机
    - 系统可以通过重试机制保证消息不会丢失

## 示例场景
以新会员注册场景为例：

1. 会员注册后需要发送相应的新会员赠送积分
2. 赠送积分在另一个服务内，需要用到分布式事务
3. 记录消息数据存为待发送状态
4. 定时任务发现待发送消息后发送到MQ
5. 积分服务消费消息并执行赠送积分的业务逻辑

## 疑问的解决 -- 为什么要设置定时任务，而不直接向MQ发送消息？
### 问题提出
有同学提出这样的疑问：业务逻辑和消息在同一个事务中执行成功后，可以直接向MQ发送消息，并在ACK回执中更新状态为已发送状态。这样能够提高消息到达的速率。而定时任务则可以充当一个兜底方案，而不是将所有发消息的操作都放到定时任务中去完成。

### 本地消息表的工作原理
让我们回顾一下本地消息表实现分布式事务的过程：

1. 将业务逻辑操作和消息发送放到一个本地事务内执行，保证业务操作和消息发送的原子性
2. 消息存入消息表内为待发送状态
3. 定时任务轮询待发送的消息列表
4. 将消息发送到MQ，下游服务消费消息，进行最终一致性操作

### 直接发送消息的问题
那么，在事务内直接发送消息到MQ会有什么问题呢？

首先，业务逻辑操作和消息的发送在一个本地事务内执行，本身就是要保证原子性，要么全部成功，要么全部失败。如果此时立即发送消息到MQ，与MQ通信超时并抛出异常，整个事务就会回滚。这意味着业务操作和消息都回滚了。

然而，MQ可能仅仅是暂时通信超时，但在MQ服务端内部可能已经接收并存储了消息。接着，下游服务订阅了该消息，就会消费该消息进行相应操作，很可能导致脏数据的产生。

### 数据库事务与MQ的独立性
我们必须清楚，数据库本地事务和MQ是两个独立的系统，它们之间无法保证强一致性。因此产生了最终一致性的解决方案。

定时任务是保证最终一致性的关键。因为如果消息存在于消息表内，那么业务操作肯定已经执行成功。此时，定时任务可以可靠地发送消息到MQ，等待下游服务消费消息。当然，可以调节定时任务的轮询频率来降低延迟。

### 正确的解决方案
如果追求立即发送消息，可以考虑基于消息系统来实现最终一致性，而不是使用本地消息表。本地消息表的设计是为了在网络抖动和MQ异常的情况下保证最终的一致性，而不是为了追求实时性。

### 结论
本地消息表的核心价值在于保障最终一致性，尤其是在网络不稳定的情况下。定时任务不仅仅是一个兜底方案，而是整个最终一致性架构中不可或缺的组成部分。

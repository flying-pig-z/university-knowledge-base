# 一.缓存使用场景
**缓存适合对一致性要求不高+高并发读取/复杂计算的场景（读取操作占比高）。**

先说一致性，缓存和数据库一般都会保证最终一致性，但是很少保证强一致性，除非加锁。但是加锁的话又会让使用缓存的优势减弱。

再说读写操作占比，缓存最主要就是读取快，就是用来读取的。

如果缓存中的数据被读取的次数寥寥无几，而且是一味的对数据进行修改，那么读取的性能开销没降低，而维护缓存的性能开销又很多，那么的话就不对了。

**更新频繁缓存命中率低是另外一回事，大不了不要使用Cache Aside策略，但是你读取操作一定要有一定占比，不然设置缓存的意义就没有了。**

另外对于复杂的查询（比如复杂 大数据量的多表查询，查一次要耗费很多时间），这时候就适合把数据放到缓存中。

# 二.读取的操作
这里想说的是不管哪种模式读操作都是一样的，这是一种统一的规范：

+ 先查询缓存，缓存命中直接返回
+ 缓存未命中，查询数据库，查询成功写入缓存

# 三.更新的操作
## 1.先更新缓存再更新数据库/先更新数据库再更新缓存
最先想到的就是上面这两种更新策略。

但是有什么问题呢？会由于并发而产生一致性问题。

比如你想想

针对于前一种情况：A进程先更新缓存-->B进程更新缓存-->B进程更新数据库-->A进程更新数据库

那这时候缓存与数据库中的数据就不一致了。

针对于后一种情况：A进程先更新数据库-->B进程更新数据库-->B进程更新缓存-->A进程更新缓存

那这时候缓存与数据库中的数据就也不一致了。**简单说就是时序性无法保证。**

**所以上面两种情况都会有并发问题。**

## 2.先更新数据库再删除缓存（Cache Aside）
### [1]为什么采用Cache Aside
为什么要在原来先更新数据库后更新缓存改为先更新数据库后删除缓存呢？

更新缓存需要每次更新数据库都更新缓存，无效写操作较多（特别是如果数据较复杂的情况），而删除缓存更新数据库时让缓存失效，查询时再更新缓存。

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1736822548481-d6344c03-b12d-4c4b-8a84-910b74dcdf56.png)

### [2]延迟双删
前面的旁路模式有个数据一致性的问题，就是说我们的键不存在了，然后去数据库查询放到缓存，刚刚查询完，然后突然数据库的数据更新并又删除了一下键，这时候执行将查询的数据放到缓存的操作，而此时放到缓存的数据是旧的数据。

此时我们可以使用延迟双删策略提高数据一致性的概率。【但不能完全解决数据一致性的问题。更加严格的数据一致性保证需要使用更复杂的机制，比如使用消息队列等】

伪代码：

```java
public void updateData(String key, Object value) {
    // 第一步：先删除缓存
    redisTemplate.delete(key);
    
    try {
        // 第二步：更新数据库
        updateDB(key, value);
        
        // 第三步：休眠一段时间（比如500ms）
        // 这个时间应该大于读取数据库并写入缓存的时间
        Thread.sleep(500);
        
        // 第四步：再次删除缓存
        redisTemplate.delete(key);
        
    } catch (Exception e) {
        // 异常处理
        log.error("更新数据失败", e);
    }
}
```

**简单说就是更新前先删除缓存，更新后睡眠一段时间后再删除缓存。**

通过延时双删策略，可以保证在数据库更新期间，其他读取请求在缓存不命中的情况下，会直接读取数据库的最新数据，而不会读取到已经失效的缓存数据。这样就保证了数据的一致性和缓存的即时更新。

延时双删策略虽然会增加一次缓存删除的开销，但是可以有效地提高数据的一致性，并且在高并发读取的场景下，减轻数据库的读取压力，提高读取性能和响应速度。

### [3]Cache Aside的问题
但是这种方法也有问题呀。

主要来自于两个方面，一致性问题和更新频率过高导致缓存命中率低的问题。

**（1）一致性问题**

比如你想想：

A进程读取不到缓存的内容读取数据库-->B进程先更新数据库-->B进程删除缓存-->A进程将数据库的数据写入缓存

**（2）更新频率过高导致缓存命中率低的问题**

**当更新频率过高的时候，缓存刚被写入数据库就被删除了，那么缓存命中率就很低**

当一个缓存命中率低于百分之九十就要特别注意。

极端点比如说秒杀场景，一秒的时间就进行几百几千次更新操作，那样的话缓存命中个der。这时候就不能更新数据库后删除缓存，而要采用最原始的先更新数据库再更新缓存或者先更新缓存再更新数据库的策略。（这里只是扣减数据，不用考虑时序性导致不一致的问题）

这里为了提高效率，一般采用的的是先更新缓存扣减库存，扣减成功再去更新数据库。

**为了进一步提高效率，我们还可以加入异步的手段。**

## 3.更新缓存然后异步的更新数据库（Write Back）
在更新数据的时候，先更新缓存，再异步批量更新数据库。

适合读多写多的操作，如果采用Cache Aside Pattern，由于更新的频繁，也频繁删除缓存。读操作如果很少命中缓存，缓存也失去了意义。

**Write Back优点是效率很高，数据库压力很小，将数据库的读和写操作多落在缓存上。(可以使用消息队列或者定时任务实现)。另外的话还可以进行异步解耦。**

**缺点：异步增大了数据库和缓存无法强一致的概率。**

比如说当过期的时候去读取，可能使得同一时间的数据更改并没有同步到缓存。一般结合前端缓存进行优化用户体验。并且我们可以在数据要过期的时候延长过期时间。如果数据一致性要求不是特别高，这个并不是特别大的问题。

**这种模式适用于对数据一致性要求不那么高的场景，比如高并发下的点赞和收藏，还有浏览量等场景。**

PS：Write-Behind 这种以cache为主的模式，是可能会丢数据的。 DB挂了，少写数据，但是只要数据还在redis能追回来 redis挂了，服务感知到后降级，请求回源，“缓存击穿”

## 4.更新数据库，然后异步的更新缓存
当然，我们也可以先更新数据库然后异步的更新缓存。

这里我们主要会使用canal监听mysql的binlog实现缓存同步。

canal通过监听数据库的binlog(比如mysql binlog)；通过binlog把数据库数据的更新操作日志（比如insert,update,delete），采集到后，通过MQ的方式，把数据同步给下游对应的消费者；下游消费者拿到数据的操作日志并拿到对应的业务数据后，再放入缓存。

![](https://cdn.nlark.com/yuque/0/2025/png/42768076/1736822548254-e818a917-8c16-405e-93ca-28e5f6165492.png)

优点：

1.把操作缓存的代码逻辑，从正常的业务逻辑里解耦出来；业务代码更加清爽和简洁，两者互不干扰和影响，独立发展。

2.速度还贼快，虽然从库到缓存经过了类canal和mq中间件，但基本上耗时都是在毫秒级，99.9%都是10毫秒内能完成库里的数据和缓存数据同步。

缺点：

1.技术方案和架构，非常复杂。中间件的运维和维护，也是个不小的工作量，所以适用于大厂而不是小厂。

2.由于引入了MQ需要解决引入MQ后带来的问题。比如数据乱序问题：同一条数据先发后至，后发先至的到达消费者后，从而引起的MQ乱序消费问题。但一般都能解决（比如通过redis lua+数据的时间戳比较方案，解决并发问题和数据乱序问题）

## 5.加锁
前面无论是先更新缓存再更新数据库，还是先更新数据库再更新缓存，还是先更新数据库再删除缓存都会出现缓存和数据库之间数据不一致。

这时候我们可以加锁保证强一致，但是这样会对性能造成损失，所以我们要做好评估。

# 四.实战总结
1.技术没有银弹，要根据实际情况选择合适的策略，不同的场景要使用不同的策略。

2.Redis中存储什么数据，存储的是数据格式也很重要。可以降低更新缓存的频率。

比如如果要查看某个用户有没有对某个音乐点赞，我们可以使用一个用户集合存储他点赞的音乐，而不是使用一个音乐集合存储它的点赞用户。

又比如我们存储的时候可以将数据分开存储，尽量把变化的部分分开，降低缓存的更新频率。

比如我们可以缓存评论信息，可以把某个内容的评论索引单独进行存储在一个集合中，而把评论的内容分开存储。这样我们先找索引再找评论内容，这样评论内容改变，就不用去更新内容的评论索引集合，而是去更新索引对应的内容缓存。

又比如我们要存储一个老师点名的学生信息集合，如果学生的分数经常变化，那我们存储学生信息集合的时候可以把学生信息中的分数字段去除，这样就不会老是更新集合。

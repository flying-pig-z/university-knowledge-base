# CAP理论
CAP理论是分布式系统设计中的一个基础理论，由Eric Brewer于2000年提出。CAP代表三个核心特性：

+ **一致性(Consistency)**: 所有节点在同一时间看到的数据是一致的
+ **可用性(Availability)**: 系统能够始终响应客户端的请求
+ **分区容错性(Partition Tolerance)**: 系统在网络分区（节点之间通信中断）的情况下仍能继续工作

CAP理论指出，在一个分布式系统中，这三个特性最多只能同时满足其中两个。也就是说，系统设计必须在这三者之间做出取舍：

+ CP系统：优先保证一致性和分区容错性，可能会牺牲可用性
+ AP系统：优先保证可用性和分区容错性，可能会牺牲强一致性
+ CA系统：优先保证一致性和可用性，但在实际的分布式环境中，网络分区是不可避免的，所以纯CA系统在实践中几乎不存在

# BASE理论
BASE理论是对CAP中一致性和可用性权衡的延伸，提供了一种更为务实的分布式系统设计思路。BASE代表：

+ **基本可用(Basically Available)**: 系统在出现故障时，允许部分功能降级，但核心功能仍然可用
+ **软状态(Soft State)**: 系统中的数据可以存在中间状态，不需要保持强一致性
+ **最终一致性(Eventually Consistent)**: 系统保证在没有新的更新操作的情况下，最终所有的访问都会返回最新的值

BASE理论是NoSQL数据库设计的理论基础，与传统关系型数据库追求的ACID特性（原子性、一致性、隔离性、持久性）形成对比。BASE更强调可用性和性能，接受数据在短时间内的不一致，但保证最终一致性。

# 两者的关系与应用
CAP理论指出了分布式系统设计的基本约束，而BASE理论则提供了在这些约束下构建实用系统的方法论：

+ 当面临网络分区时，系统可以选择牺牲强一致性（遵循BASE原则），以保持系统的可用性
+ 大型互联网应用通常选择AP+最终一致性的模式，如Amazon的Dynamo、Cassandra等
+ 金融、银行等领域则可能更倾向于CP模式，如Google的BigTable、HBase等

在实际应用中，系统设计往往是在一致性和可用性之间找到适合特定业务场景的平衡点，而不是简单地二选一。



> 分布式系统事务和本地事务不同，强一致性很难保证，通常取最终一致性。

### 一、背景：单体 Agent 的天花板
单个 Agent 再强，同时能处理的信息量是有限的——它本质上是一个单体智能。复杂任务靠单体 Agent 就像靠一个人解决所有问题，效率和质量都有瓶颈。下一步进化方向：**多 Agent 协同**。

---

### 二、架构演进：从单 Agent 到 Swarm
#### 阶段 1 · 单体 Agent（串行）
一个窗口，任务依次执行。问题：**上下文越来越长**，效率低。

#### 阶段 2 · 手动多窗口并行
人工把任务拆开，分别在多个窗口执行。问题：**各窗口代码改动冲突需要人工协调**。

#### 阶段 3 · Sub Agents（主控子代）
一个主 Agent 自动开多个子 Agent 并行执行，最后汇总结果。子 Agent 将执行结果总结后返回给主 Agent，**上下文长度可控**。

局限：子 Agent 之间**不能直接通信**，只适合改动相对独立、耦合少的任务。

#### 阶段 4 · Agent Teams（共享上下文）
主 Agent 通过**共享任务清单**分配任务，各子 Agent 可**互相通信**，协作更紧密。

两个严重问题：

+ **Token 爆炸**：所有子 Agent 共享全部历史消息，任务越往后消耗越大
+ **缺乏弹性**：成员在任务开始时固定，无法中途动态扩容

#### 阶段 5 · Agent Swarm（动态编排）
Kimi Agent Swarm 的方案，解决了上述两个问题：

| 特性 | 做法 | 效果 |
| --- | --- | --- |
| **上下文隔离** | 每个子 Agent 只持有自己负责部分的上下文 | Token 大幅降低，减少幻觉和出错 |
| **动态并行** | 主 Agent 根据任务量现场决定创建几个子 Agent | 任务量大时自动裂变，效率提升 4 倍以上 |
| **局部容错** | 单个 Agent 报错不影响整体，内部自动重试 | 系统稳定性更高 |


技术支撑：Kimi K2.5 在训练时引入了 **PARL（并行代理强化学习）**，使其天然具备动态编排能力。

### 三、实测案例：硅基文明简史网页
任务：整理 100 个推动 AI 发展的历史事件，生成风格一致的图片，最终制作成网页。

Kimi Agent Swarm 的执行流程：

1. **研究员 Agent** → 多轮联网搜索，收集 138 个候选事件，输出 MD 文档
2. **审核员 Agent** → 淘汰 38 个，留下 100 个，输出筛选结果
3. **图片生成员 Agent** → 发现 100 张图任务量大，**动态裂变出 5 个子 Agent 并行生图**
4. **图片审核员 Agent** → 校验风格一致性
5. **UI 设计师 Agent** → 定义网页风格
6. **程序员 Agent** → 编写并运行网页代码

最终几分钟内完成了人工需要数天才能完成的任务。

### 四、Agent Teams vs Agent Swarm 核心对比
| 维度 | Agent Teams（Claude Code） | Agent Swarm（Kimi） |
| --- | --- | --- |
| **上下文策略** | 所有 Agent 共享全部历史 | 每个 Agent 上下文物理隔离 |
| **Token 消耗** | 高，随任务累积爆炸 | 低，只处理局部 |
| **动态扩容** | 成员开始时固定 | 可根据任务量现场裂变 |
| **容错能力** | 局部失败影响全局 | 局部失败不影响整体 |
| **集成度** | 专业工具，需自配环境和 Skills | 自带虚拟机，生图/搜索/代码开箱即用 |
| **适用人群** | 开发者 | 普通用户友好 |

### 五、核心洞察
从单体智能到群体协作，AI 正在复刻人类文明的跃迁路径——不是更聪明的大脑，而是更聪明的协作。

Agent 集群，可能才是通往通用人工智能（AGI）的真正路径。

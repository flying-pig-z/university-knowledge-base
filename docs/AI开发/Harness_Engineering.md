## 一、三个概念的总览关系
2026 年 OpenAI 在博客中提出了 **Harness Engineering（驾驭工程）** 的概念，迅速在 AI 圈走红。它与此前流行的 **Prompt Engineering（提示词工程）** 和 **Context Engineering（上下文工程）** 构成了一个层层递进的体系：

+ **提示词工程** → 解决大模型"无引导乱说话"的问题
+ **上下文工程** → 解决上下文的组织与管理问题
+ **驾驭工程** → 解决 AI Agent 持续、规范地执行任务并最终交付的问题

三者的包含关系：**提示词 ⊂ 上下文 ⊂ Harness**，提示词工程是上下文工程的一部分，上下文工程又是驾驭工程的一部分。

---

## 二、第一层：Prompt Engineering（提示词工程）
### 大模型的本质
把 ChatGPT / Claude 的外壳剥开，里面的 LLM 本质上是一个磁盘上的**超大参数文件**：

+ 加载到显卡内存 + HTTP 接口 → 大模型 API 服务
+ 加上聊天界面 → 聊天 AI
+ 加上代码编辑器 → AI IDE

大模型做的事情很简单：**基于当前输入内容，预测下一个字词大概率是什么**。它本质上只是在"猜"你想要什么。

### 为什么需要提示词工程
如果输入的指令太宽泛，模型预测的答案就会非常发散。例如丢一段代码说"加个排序"，它可能只给排序片段；补上"给我完整函数代码，不要乱改我的代码"后，结果才更符合要求。

### 可加入的提示词要素
角色设定、背景信息、历史对话、参考文档、输出格式限制……这些约束统称为**提示词**。

### 定义
**提示词工程**：有意识地调整和设计提示词，让模型稳定地朝着预期的内容和格式输出的技术手段。

## 三、第二层：Context Engineering（上下文工程）
### 从提示词到上下文
提示词写得越长越仔细，模型知道的就越多，回答就越准。大家很自然地不断往模型里塞各种资料——这些打包在一起发给大模型的**所有信息**就叫**上下文**。提示词只是上下文的一部分。

### 上下文窗口与上下文腐化
+ **上下文窗口**：大模型一次性能处理的上下文最大限制。
+ 多轮对话容易把窗口打满，需要压缩或丢弃部分信息。
+ 这个过程中不可避免会丢失关键信息，破坏上下文的完整性和准确性，此类问题统称为**上下文腐化**。
+ 表现：模型开始记不住、回答前后不一致。

### 核心问题
怎么才能在合适的时候，将合适的内容塞入有限的上下文中？

### 上下文工程三步骤
以 Cursor、Claude Code、Trae 等 Coding Agent 为例，虽然各家实现有差异，但总体可归纳为：

| 步骤 | 说明 |
| --- | --- |
| **① 召回** | 从外部新闻、聊天记录、当前代码环境、运行报错等来源中，找出最相关的内容（涉及 RAG、Memory 等技术） |
| **② 压缩** | 上下文窗口有限，需要将信息变小，例如分段发给大模型做总结 |
| **③ 组装** | 信息的位置和顺序直接影响模型理解（越靠后越容易被关注），需要通过一定结构重新组装内容 |


### 为什么同一模型在不同工具中效果不同
不同 AI 工具的上下文工程策略不同。即使底层模型相同，组装出的上下文质量也不同，因此执行效果存在差异。

## 四、AI Agent 与 ReAct 循环
### 执行层的诞生
模型再聪明也只能聊天，无法帮我们"干活"。给大模型加入以下能力，让它像人一样操作外部工具：

+ Bash 沙箱
+ 文件系统读写
+ MCP（Model Context Protocol）
+ ……

这些能力共同构成了**执行层**。

### ReAct 循环
将上述能力串成一个流程，外部套一层循环：

```plain
提示词/上下文工程 → 组装上下文 → 大模型思考 → 外部程序执行 → 执行结果回写上下文 → 继续推理
```

这套"一边思考一边行动"的循环就是 **ReAct**。而这个能通过聊天帮你执行任务的程序，就是 **AI Agent**。

**Agent 的本质就是一个 for 循环。**

---

## 五、第三层：Harness Engineering（驾驭工程）
### 为什么需要驾驭工程
Agent 循环一长，上下文必然膨胀，即使上下文工程做得再好也可能腐化。随着 Agent 看过的文件越来越多、信息越来越杂，前面定好的目标和约束会逐渐被"冲淡"，理解越来越偏。

### 解决思路：规则文件（记忆层）
只要保证每次给大模型的上下文中，都包含一些**可复用的核心信息**——项目目标、技术栈、需求背景、代码风格、禁止事项等——大模型就能在大框架约束下减少理解偏移。

这些核心信息可以单独写成文件，固定在代码仓库里：

| 工具 | 规则文件 |
| --- | --- |
| Claude Code | `CLAUDE.md` |
| Cursor | Rules 文件 |
| Trae | 各自的 Rules 文件 |


规则文件在调用大模型时，作为**系统提示词自动注入上下文**。

文件过长时可以拆分 + 路由，例如背景读 `bg.md`，技术栈看 `stack.md`，一般只加载文件路径，真正需要时才加载全部内容。

### 驾驭工程的四层架构
| 层级 | 职责 | 说明 |
| --- | --- | --- |
| **编排层** | 全局规划与任务拆解 | 将大任务拆解为有明确执行标准的多个子任务，按规划驱动 Agent 分步执行，避免跑偏或陷入死循环 |
| **记忆层** | 核心信息持久化 | 规则文件 + 提示词工程 + 上下文工程配合，保证关键约束始终在上下文中 |
| **执行层** | 外部工具操作 | Bash 沙箱、文件系统、MCP 等，让 Agent 能读写代码、执行命令、做测试 |
| **反馈层** | 校验与自动修复 | 将测试输出和报错加入上下文，驱动 Agent 在下一轮循环中自动修复问题 |


### 核心公式
**Agent = 大模型 + Harness**

只要不是大模型本身的部分，都属于 Harness Engineering 的范畴。

大模型越强，外壳可以做得越薄，但**无论如何这层外壳都必须有**。

## 六、落地实践：以 Claude Code 为例
### 最轻量的做法
在 `CLAUDE.md` 文件里写清楚：

1. 项目背景是什么
2. 希望大模型做什么、不做什么
3. 做完后要跑哪些 Lint、单测和 CI
4. 执行哪些 Script

### 进阶：Spec Driven Development（SDD）
引入 Spec Kid 等扩展插件，根据项目将需求拆成多个阶段：

1. **生成约束文件** → 明确需求
2. **制定开发计划** → 拆解任务
3. **实际修改 + 测试** → 执行交付

每个阶段都可能更新一次 `CLAUDE.md`，确保每一阶段注入上下文的尽可能都是核心信息。这套开发方式本质上就是 Harness Engineering 的落地。

---

## 七、总结
| 工程层级 | 解决的问题 | 一句话概括 |
| --- | --- | --- |
| **提示词工程** | 大模型无引导乱说话 | 让大模型明白你的具体需求和输出标准 |
| **上下文工程** | 上下文的组织与管理 | 给大模型注入精准有效的上下文 |
| **驾驭工程** | Agent 持续规范执行 | 让大模型持续按规范执行任务，并最终交付 |


程序员的工作内容正在从"写代码"转向"写规则和 Skill"——Harness Engineering 将成为 AI 时代的主战场。

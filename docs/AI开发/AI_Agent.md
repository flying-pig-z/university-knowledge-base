## 二、推理服务：大模型是怎么工作的
ChatGPT、DeepSeek 这类大模型，本质上就是一个**超大参数文件**，躺在磁盘上，里面装的是训练时学到的知识参数。

要让它工作，需要一个程序把它**加载到内存**，对外暴露 HTTP 接口，接收用户请求、做推理、返回结果——这就是**推理服务**。

给推理服务配上前端网页聊天框，就成了我们熟悉的聊天 AI。

---

## 三、Memory（记忆机制）
### 问题：大模型本身什么都不记得
推理服务本质是 HTTP 服务，每个请求处理完就结束，不保存任何状态。而且为了抗高并发，一般会部署多个实例做负载均衡——第一次请求打到机器 A，第二次可能打到机器 B，完全是不同的进程。

### 怎么做到"看起来有记忆"
每次请求时，系统把之前的聊天记录**重新拼到对话里**，一起发给大模型。这些拼起来的内容统称**上下文**。大模型看到完整上下文，自然就能接上话。

### 上下文过长怎么办？分两类管理
| 类型 | 做法 |
| --- | --- |
| **短期记忆** | 当前会话中最近几轮对话完整保存 |
| **长期记忆** | 很久之前的对话提取关键信息，压缩成摘要 |


每次请求时将短期 + 长期记忆拼成对话发给大模型，大模型看起来就像有记忆一样。

**Memory**：这套管理上下文的机制。

---

## 四、RAG（检索增强生成）
### 问题：大模型的知识是固定的
大模型的训练数据来自互联网历史公开数据，训练完成后知识就固定了。你问它今天的新闻或公司内部文档，它根本不可能知道。

### 解决方案：外部知识库
给大模型配一个外部知识库（最新新闻、公司内部文档等）。用户提问时：

1. 先从知识库中**检索**匹配的相关知识
2. 将检索结果与用户问题一起**喂给大模型**
3. 大模型基于外部知识**生成**回答

**RAG**（Retrieval Augmented Generation）：检索外部知识 → 增强 → 生成回答。

### 向量数据库解决语义匹配
传统数据库只能做字面匹配，无法理解语义相似性。解决方案是把文本转成**向量**，用向量距离衡量语义相似度，语义相近的文本就能匹配上。

RAG 用的数据库里存的是向量数据，这种数据库叫**向量数据库**（如 Milvus）。数据量不大时也可以用 PostgreSQL（pgvector 扩展）。

## 五、MCP（模型上下文协议）
### 问题：大模型只能对话和思考，像"缸中大脑"没有手脚
### 解决方案：约定一种工具调用的消息格式
1. 外部先告诉大模型**有哪些工具可用**及其格式
2. 大模型想用工具时，输出一段**特定格式的 JSON**（如发邮件：写清楚发给谁、发什么）
3. 外部收到后**执行操作**，将结果返回给大模型
4. 大模型基于工具执行结果**生成最终回复**

**MCP**（Model Context Protocol）：让大模型通过结构化消息调用外部工具的协议规范。

### MCP 的组成结构
| 概念 | 说明 |
| --- | --- |
| **MCP Host** | 在外部负责解析 JSON 并操作工具的程序（如 Cursor、Claude Code） |
| **MCP 插件** | 能被调用的具体工具 |
| **MCP Client** | 插件的本地部分，负责接收调用请求 |
| **MCP Server** | 插件的远端部分，部署在服务提供方的服务器上，真正执行 API 操作（如 GitHub MCP Server 部署在 GitHub 服务器上） |

## 六、Skills（操作技能）
### 问题：有了工具，但不知道怎么用
MCP 给了大模型一堆工具，但大模型不一定知道该**按什么顺序用、怎么组合用**。就好比给大学生一堆钳子扳手，他也不一定能修好车——他缺的是**经验和流程**。

### 解决方案：写一份操作手册
一份结构化的操作指南，详细说明：遇到什么场景、用什么工具、先做什么后做什么、有什么注意事项。

**Skills**：规定在什么场景下，按什么顺序组合使用哪些工具的操作经验。

### 示例：排查线上事故
| 层级 | 提供的能力 |
| --- | --- |
| **MCP** | 查监控、查日志、查配置、回滚版本等工具能力 |
| **Skills** | 规定流程：先看监控判断影响范围 → 查日志和配置定位模块 → 必要时执行回滚 |


### 一个类比
+ 大模型 = **大脑**（思考和规划）
+ MCP 协议 = 让大脑有了**手**
+ MCP 插件 = 手上的**工具**
+ Skills = **操作经验**

---

## 七、AI Agent（智能代理）
### 定义
大模型本来就能思考和规划，在此基础上叠加：

| 能力 | 作用 |
| --- | --- |
| Memory | 记住历史对话 |
| RAG | 获取外部知识 |
| MCP | 操作外部工具 |
| Skills | 知道怎么组合使用工具 |


它们共同构成了一个在某些功能上能**代替人类自主行动、完成目标**的 AI 系统，即 **AI Agent**。

Agent 本质上就是一个**智能工具人**。通过提示词设定角色，它可以是智能客服、程序员、私人律师等各种角色。

---

## 八、实例：Claude Board 与 Manus 的对比
| 维度 | Claude Board | Manus |
| --- | --- | --- |
| 本质 | 帮你自动操作电脑的 AI Agent | 同类产品 |
| 运行环境 | 主要面向**本地电脑** | 考虑安全问题，将操作环境放到**远端虚拟机** |
| 能力范围 | 你能用电脑干什么，它就能干什么（发邮件、投简历、做交易等） | 类似 |
| 安全性 | 开源，权限安全需用户自行考虑 | 远端沙箱隔离，相对更安全 |
| 技术突破 | 没有本质技术突破，但开源带来的自由度让它迅速走红 | — |


如果将 Claude Board 部署到远端服务器上，就有点"开源版 Manus"的味道。

## 九、总结
```plain
推理服务（大模型本体）
    │
    ├── + Memory  → 记住历史，管理上下文
    ├── + RAG     → 检索外部知识，突破训练数据限制
    ├── + MCP     → 调用外部工具，从"只能聊"到"能动手"
    ├── + Skills  → 操作经验，知道工具怎么组合使用
    │
    └── = AI Agent → 自主行动、完成目标的智能系统
```
